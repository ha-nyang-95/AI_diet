## Story

As a **엔드유저**,
I want **일별 식단 기록을 한 화면에서 시간순으로 보기를**,
So that **오늘의 식사·매크로·피드백 요약·fit_score를 한눈에 확인하고 다음 끼를 결정할 수 있다.**

본 스토리는 Epic 2의 네 번째 스토리로, **(a) `(tabs)/meals/index` 일별 기록 화면 polish — 날짜 picker(prev/today/next + "오늘로 이동" CTA) + 시간순(아침→저녁 ASC) 정렬 + 임의 일자 조회**, **(b) `MealCard.tsx` polish — 사진 썸네일(`expo-image`) + `parsed_items` 부제(DF15 해소) + Epic 3 분석 슬롯(`analysis_summary` forward-compat — 매크로/피드백 1줄/fit_score 색상+숫자+텍스트 라벨, NFR-A4) + "AI 분석 대기" placeholder**, **(c) 백엔드 `GET /v1/meals` 날짜 필터 W50 해소 — `Asia/Seoul` TZ 강제 적용으로 KST 0-9시 식단 누락 차단(D1 결정)**, **(d) `(tabs)/meals/[meal_id].tsx` 상세 placeholder 페이지 신규 — meal 텍스트/이미지/parsed_items + "AI 분석은 Epic 3에서 도입 예정" placeholder(Story 3.7가 react-native-sse 채팅 wire)**, **(e) 7일 로컬 캐시(NFR-R4) — TanStack Query `@tanstack/react-query-persist-client` + `@tanstack/query-async-storage-persister` AsyncStorage 24h gc + 7d maxAge — 오프라인/DB 다운 시 마지막 7일 읽기 전용 표시 + 네트워크 상태 배너**, **(f) `MealResponse.analysis_summary` 슬롯 인터페이스 정의 — Epic 3 `meal_analyses` 테이블과 1:1 매핑 forward-compat — Story 2.4 baseline은 항상 null(서버 join X)**, **(g) NFR-A3 폰트 스케일 200% 회귀 검증 — 카드/날짜 picker/[meal_id].tsx 모두 `flexShrink + flexWrap + lineHeight` 보장**의 baseline을 박는다.

부수적으로 (a) `meal_analyses` 테이블 + `analysis_summary` 실제 wire(JOIN, fit_score 계산)는 **Story 3.3 / 3.5 책임** — 본 스토리는 *인터페이스 슬롯 + UI placeholder*만, (b) `[meal_id].tsx` SSE 채팅 통합은 **Story 3.7 책임** — 본 스토리는 *meal 상세 + 분석 placeholder + 라우팅 진입점*만, (c) 오프라인 텍스트 큐잉은 **Story 2.5 책임** — 본 스토리는 *읽기 캐시*만(쓰기는 Story 2.5), (d) `parsed_items` 음식명 정규화(*짜장면/자장면*)는 **Story 3.4 책임** — 본 스토리는 *raw 표시*만, (e) 알레르기 위반 시 fit_score 0점 단락은 **Story 3.5 책임** — 본 스토리는 fit_score 표시 슬롯만, (f) jsonb GIN 인덱스(검색)는 **Story 8 책임** — 본 스토리는 *날짜 필터*만, (g) 다중 TZ 지원(외주 다국가)은 **Growth NFR-L4 책임** — 본 스토리는 *Asia/Seoul 강제*(NFR-L2 한국어 1차 정합), (h) cursor pagination은 **Story 8 책임** — 본 스토리는 *limit=200 + 1일 단위 조회*. 본 스토리 종료 시 Epic 2의 일별 기록 UI baseline 완료 → Story 2.5(오프라인 텍스트 큐잉)이 *쓰기 캐시* 추가, Story 3.x가 `analysis_summary` 슬롯 wire, Story 3.7이 `[meal_id].tsx` SSE 통합으로 자연 진화.

## Acceptance Criteria

> **BDD 형식 — 각 AC는 독립적 검증 가능. 모든 AC 통과 후 `Status: review`로 전환하고 code-review 워크플로우로 진입.**

1. **AC1 — `GET /v1/meals` 날짜 필터 `Asia/Seoul` TZ 강제 (W50 해소, D1 결정)**
   **Given** Story 2.1 baseline 라우터 + W50 deferred(KST/UTC drift), **When** 클라이언트가 `?from_date=2026-04-30&to_date=2026-04-30` 송신, **Then**:
     - 서버는 `from_date`/`to_date`를 *`Asia/Seoul` TZ 자정 boundary*로 해석. 예: `from_date=2026-04-30` → `'2026-04-30 00:00:00+09:00'` 이상, `to_date=2026-04-30` → `'2026-05-01 00:00:00+09:00'` 미만.
     - SQL 변경: `Meal.ate_at >= from_date` (현재) → `Meal.ate_at >= func.timezone('Asia/Seoul', from_date_with_tz)` 또는 더 간단히 *date → timestamptz 변환을 Python 측에서 명시적으로 처리* (Postgres `AT TIME ZONE` 회피 — driver 호환성). Python 변환:
       ```python
       from zoneinfo import ZoneInfo
       _KST = ZoneInfo("Asia/Seoul")
       if from_date is not None:
           from_dt = datetime.combine(from_date, time.min, tzinfo=_KST)
           stmt = stmt.where(Meal.ate_at >= from_dt)
       if to_date is not None:
           to_dt = datetime.combine(to_date + timedelta(days=1), time.min, tzinfo=_KST)
           stmt = stmt.where(Meal.ate_at < to_dt)
       ```
     - **D1 결정 (W50 해소)**: `Asia/Seoul` 강제 — 다중 TZ 지원(외주 다국가)은 **Growth NFR-L4** (option C "클라이언트 UTC 강제" 회피 사유: 모바일 wire 단순성 우선 + NFR-L2 한국어 1차 정합 + KST/UTC 변환은 백엔드 단일 책임).
     - **회귀 차단 단언**: KST 0-9시 입력 식단(예: `ate_at = 2026-04-30T03:00:00+09:00` → UTC `2026-04-29T18:00:00Z`)이 `from_date=2026-04-30&to_date=2026-04-30` 쿼리에 *포함*되어야 함. 현 동작은 *제외* (W50 핵심 결함).
     - **`from_date` 단독**, **`to_date` 단독**, **둘 다 None** 케이스 모두 동작. None인 쪽은 boundary 미적용.
     - `cursor` 비-null 거부는 그대로 유지 (D2 / Story 2.1).
     - **OpenAPI 변경 X** — wire format 동일(`date` ISO 문자열), 의미만 KST로 명료화. OpenAPI description 1줄 갱신: *"from_date/to_date는 `Asia/Seoul` TZ의 자정 boundary로 해석됩니다."*

4. **AC4 — `mobile/app/(tabs)/meals/index.tsx` polish — 날짜 picker + 시간순 ASC 정렬 + 임의 일자 조회**
   **Given** Story 2.1 baseline 일별 기록 화면 (오늘 only) + Story 2.4 일자 picker, **When** 사용자가 화면 진입, **Then**:

       - 헤더 title: *"오늘 식단"* (오늘 일자 시) 또는 *"4월 30일 식단"*(다른 일자 시 — `Intl.DateTimeFormat('ko-KR', { month: 'long', day: 'numeric', timeZone: 'Asia/Seoul' })`).

       - 헤더 right area: `>` (다음 일자) — `onPress` → `setSelectedDate(prev => addDays(prev, 1))`. 현재 일자가 *오늘 + 1일* 미만(즉, *내일까지 허용*)이면 disabled — *오늘 이후*는 미래 시점 식단 입력 흐름이라 *조회 limit*은 *내일 자정까지*(catch-up 시나리오 — Story 2.1 정합).
       - "오늘로 이동" 버튼: 현재 일자가 *오늘이 아닐 때만* 노출 (`selectedDate !== todayKst()` 시). 버튼은 헤더 right(`>` 옆) 또는 화면 상단 작은 secondary CTA.
       - "+" 식단 입력 버튼은 그대로 유지 (Story 2.1 정합) — 헤더 right 끝.
     - **시간순 ASC 정렬 (epic AC line 558 — *"시간(아침→저녁)순"*)**: `useMealsQuery({ from_date, to_date })` 응답을 *클라이언트 측 정렬*하지 말고 *서버 ORDER BY* 변경 — Story 2.1의 `Meal.ate_at.desc()` → `Meal.ate_at.asc()`. **그러나** desc 정렬은 *오늘 식단 빠른 조회*에 유리(가장 최근 식단을 위에 — 새로 입력한 식단을 즉시 확인). **D2 결정**: *날짜 필터 활성 시 ASC 강제*(특정 일자 → 시간순). *날짜 필터 미적용 시*(전체 조회 — Story 8 cursor pagination)는 DESC 유지. 본 스토리는 *날짜 필터 항상 적용*이므로 ASC만 노출.
       - 백엔드 `list_meals` ORDER BY: `if from_date is not None or to_date is not None: order_by = Meal.ate_at.asc() else: order_by = Meal.ate_at.desc()`. 날짜 필터 일관 활성화 케이스에 대비.
       - 본 결정은 deferred-work에 D2로 명시 — *"전체 조회 시 정렬*은 Story 8 cursor pagination 시점에 재검토(아래 신규 deferred 후보 참조).
     - **selectedDate state**:
       - 초기값: `todayKst()` — `new Date()`를 `Asia/Seoul` TZ 자정 boundary 로 정규화. helper `mobile/features/meals/dateUtils.ts` 신규.
       - 형식: `string` (`YYYY-MM-DD` KST) — `useMealsQuery({ from_date: selectedDate, to_date: selectedDate })`로 forward.
     - **빈 일자 화면**:
       - `selectedDate === todayKst()` 시: *"오늘 기록 없음 — 첫 식단을 입력하세요"* + CTA *"식단 입력"* (Story 2.1 정합).
       - `selectedDate !== todayKst()` 시: *"${formatDateKo(selectedDate)} 기록 없음"* (CTA 미노출 — 과거 일자 시점에 입력 시 *어제 식단 입력*은 input.tsx에서 `ate_at` 명시 송신으로 처리, 일별 기록 화면 자체는 *조회 only*).
     - **카드 onPress 흐름 신규** (AC3 정합): `onCardPress(meal) → router.push(\`/(tabs)/meals/${meal.id}\`)`. typed-routes cast은 Story 2.1 W9 정합 패턴 사용 (`as Parameters<typeof router.push>[0]`).
     - **long-press → ActionSheet** 흐름은 Story 2.1 그대로(edit/delete) — 회귀 0건.

### Architecture decisions (반드시 따를 것)

[Source: architecture.md#Data Architecture (line 281-299), architecture.md#API Patterns (line 314-326, 455-482), architecture.md#Architectural Boundaries (line 845-868), architecture.md#Project Structure (line 634-700, 800-833), architecture.md#Naming Patterns (line 386-431), epics.md#Story 2.4 (line 550-563), prd.md#FR14, prd.md#NFR-A3, prd.md#NFR-A4, prd.md#NFR-R4, prd.md#NFR-L2, Story 2.1/2.2/2.3 패턴]

**`Asia/Seoul` TZ 강제 (W50 해소, D1 결정)**

- 백엔드 `GET /v1/meals` 날짜 필터를 `Asia/Seoul` TZ 자정 boundary로 해석. KST 0-9시 식단 누락 차단(`ate_at='2026-04-30T03:00:00+09:00'` UTC `2026-04-29T18:00:00Z`이 `from_date=2026-04-30`에 포함).
- 구현 — `from zoneinfo import ZoneInfo` + `_KST = ZoneInfo("Asia/Seoul")` 모듈 상수 + `datetime.combine(from_date, time.min, tzinfo=_KST)` Python-측 변환. Postgres `AT TIME ZONE` 회피 — driver 호환성(asyncpg) + Python 명시성 우선.
- D1 결정 사유 (option A/B/C 중 B 선택):
  - **B(채택) — 서버 `Asia/Seoul` 강제**: NFR-L2 한국어 1차 정합 + 클라이언트 wire 단순(`from_date=YYYY-MM-DD` 그대로) + 백엔드 단일 책임 (TZ 변환 로직 1지점). 다중 TZ 지원은 Growth NFR-L4.
  - **A — `tz` 쿼리 파라미터**: 외주 클라이언트 다국가 시점 유연성 ↑이나 본 MVP에서 over-engineering(외주 인수 클라이언트가 한국 1차 가정).
  - **C — 클라이언트 UTC 강제**: 모바일 `Date.toISOString()` UTC 송신 + 서버 timestamp 범위 그대로 — but `to_date+1day` boundary 의미가 모바일 클라이언트 책임으로 분산 + W49(formatHHmm) 정합 어려움.
- Postgres 세션 TZ는 default `UTC` 유지 — `meals.ate_at` `timestamptz` 컬럼은 항상 UTC 저장(asyncpg 표준), 쿼리 시 KST 명시 변환만 적용.
- Sub-effect: `list_meals` ORDER BY 분기(D2) — 날짜 필터 활성 시 ASC(시간순 — 아침→저녁), 미적용 시 DESC(최근 식단 우선 — Story 8 cursor pagination forward). 본 스토리는 *날짜 필터 항상 활성*이므로 ASC만 노출.

### Anti-pattern 카탈로그 (본 스토리 영역)

[Source: Story 2.1/2.2/2.3 Anti-pattern 카탈로그, architecture.md#Anti-pattern (line 557-565)]
