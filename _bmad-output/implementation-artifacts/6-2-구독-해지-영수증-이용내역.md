## Story

As a 엔드유저,
I want 구독을 언제든 해지하고 자기 영수증·이용내역을 조회하기를,
So that 결제 투명성 + 사용자 권리(약관 명시 해지권)를 보장받는다.

## Acceptance Criteria

1. **AC1 — `POST /v1/payments/cancel` endpoint 신설 + DB-only `status=cancelled` + `cancelled_at` set + cancel audit `payment_logs` row 1건**
   **Given** epics.md:852 *"`POST /v1/payments/cancel` → `subscriptions.status=cancelled` + `cancelled_at` 기록 + 다음 결제일까지 활성 (즉시 차단 X — 표준 정기결제 패턴)"* + Story 6.1 baseline은 *1회 결제만*(billing key 미발급 → 자동 갱신 없음) + Story 5.x `Depends(current_user)` PIPA Art.35 패턴, **When** 사용자가 해지 액션, **Then**:
   - **신규 endpoint** `POST /v1/payments/cancel` — `api/app/api/v1/payments.py`에 추가:
     - `status_code=200` (해지는 *상태 전이* — 신규 리소스 생성 아님이라 201 X. RFC 9110 정합).
     - **Pydantic 모델 신규**:
       - `CancelSubscriptionRequest(BaseModel)` — `model_config=ConfigDict(extra="forbid")` + 본문 0필드(*명시 확인 의도는 클라이언트 UI에서 처리, 본문은 빈 dict {} 허용*). 향후 사유(`reason: str`) 추가 시 forward-compat.
       - `CancelSubscriptionResponse(BaseModel)`: `subscription: SubscriptionResponse` + `payment: PaymentLogResponse` (cancel audit row).
     - **endpoint 시그니처**:
       ```python
       @router.post(
           "/cancel",
           status_code=status.HTTP_200_OK,
           response_model=CancelSubscriptionResponse,
           responses={
               200: {"description": "Subscription cancelled (active until expires_at)"},
               404: {"description": "No active subscription"},
               409: {"description": "Subscription already cancelled or expired"},
           },
       )
       async def cancel_subscription(
           db: DbSession,
           user: Annotated[User, Depends(current_user)],
           body: CancelSubscriptionRequest = CancelSubscriptionRequest(),
       ) -> CancelSubscriptionResponse: ...
       ```
     - **Dependency**: `Depends(current_user)` 단독 — `Depends(require_basic_consents)` *미적용*. **사유**: PIPA Art.35 정보주체 권리 + *해지권은 약관 명시 사용자 권리*(epics.md:847 *"약관 명시 해지권"* 정합) — 동의 철회와 *독립*된 권리. `get_subscription`(6.1)와 동일 dependency 정합.
   - **service 함수 신설** `payment_service.cancel_subscription(...)`:
     - `async def cancel_subscription(session: AsyncSession, *, user_id: uuid.UUID) -> tuple[Subscription, PaymentLog]`.
     - **Step 1 — active 또는 cancelled-but-not-expired subscription fetch**:
       - `select(Subscription).where(Subscription.user_id == user_id, Subscription.status == 'active')` 1차 조회.
       - 결과 None → `NoActiveSubscriptionError(404)` raise(*"활성 구독이 없습니다 — 신청 후 해지할 수 있습니다"*).
     - **Step 2 — DB UPDATE**:
       - `subscription.status = "cancelled"` + `subscription.cancelled_at = datetime.now(UTC)` + `subscription.updated_at`은 `onupdate=now()` server-side 자동 set.
       - **`expires_at`은 변경 X** — epics.md:852 *"다음 결제일까지 활성"* 정합. 사용자는 `expires_at`까지 모든 기능 정상 이용. webhook(Story 6.3)이 `expires_at` 도래 시 `status='expired'` 전이.
     - **Step 3 — cancel audit `payment_logs` row 1건 INSERT**:
       - `event_type="cancel"` + `provider="toss"`(향후 stripe 분기는 `subscription.provider` 정합) + `provider_payment_key=NULL`(cancel은 PG 호출 X) + `provider_order_id=NULL` + `amount_krw=0`(*환불 0* — 다음 결제일까지 활성 정합 — 즉시 환불은 OUT) + `status="success"` + `idempotency_key=NULL`(본 스토리는 cancel idempotency 불필요 — 이미 cancelled 상태 재호출은 `SubscriptionAlreadyCancelledError(409)` 가드) + `raw_payload={"reason": "user_initiated"}` + `occurred_at=now()` + `subscription_id=subscription.id`.
       - 향후 사유 캡처(`CancelSubscriptionRequest.reason`) 시 `raw_payload["user_reason"]` field로 추가.
     - **Step 4 — Returns** `(subscription, cancel_log)`. commit은 라우터 책임(`subscribe`(6.1) 패턴 정합).
   - **흐름 정합**:
     1. `subscription, cancel_log = await payment_service.cancel_subscription(db, user_id=user.id)`.
     2. `await db.commit()`.
     3. `return CancelSubscriptionResponse(subscription=_subscription_to_response(subscription), payment=_payment_log_to_response(cancel_log))`.
   - **Toss API cancel 호출 X 결정**:
     - Toss `POST /v1/payments/{paymentKey}/cancel` API는 *환불* 트리거(*전체/부분 환불*) — *"다음 결제일까지 활성"* invariant와 충돌(즉시 환불 시 사용자 사용 권리 박탈).
     - Story 6.1 baseline은 *(a) 결제 위젯 패턴* — *최초 1회 결제만* 진행, billing key 미발급, *자동 갱신 자체가 없음*. 따라서 cancel은 *DB 상태 전이*만으로 충분 — 다음 결제 주기는 *원래부터* 도래하지 않음.
     - 환불 자동화는 epics.md:825 OUT(*"PG 본 계약·세금정산은 OUT"*).
     - Story 6.3 webhook이 도입되면 *renew 결제* 시점에 *cancelled 상태 가드*로 갱신 차단(*expires_at 도래 → status=expired*).
   - **Sentry**: 명시 capture X — `cancel`은 *정상 사용자 액션*(에러 조건 아님). `logger.info("payments.cancel.requested", user_id=_mask_user_id(user.id), subscription_id=str(sub.id))` + `logger.info("payments.cancel.succeeded", ...)` 운영 모니터링.
   - **rate limit**: default `["1000/hour", "60/minute"]` 자동 적용. 추가 데코레이터 X(분당 60회 cancel은 abuse 신호 — default cap 충분).

2. **AC2 — `cancelled-but-not-expired` UX contract — `GET /v1/payments/subscription`이 cancel 후에도 200 반환 (Story 6.1 deferred 흡수)**
   **Given** Story 6.1 deferred(`cancelled-but-not-expired` UX gap — `get_subscription` 404 반환) + epics.md:852 *"다음 결제일까지 활성"* invariant + 사용자 *"내 구독 상태"* 의도(*active까지 며칠 남았는지 확인*), **When** 사용자가 cancel 후 `GET /v1/payments/subscription` 호출, **Then**:
   - **`get_subscription` endpoint(6.1) contract 변경** — `api/app/api/v1/payments.py:get_subscription`:
     - **이전(6.1)**: `Subscription.status == 'active'` 1차 조회 → None이면 404.
     - **신규(6.2)**: `Subscription.status.in_(['active', 'cancelled'])` AND `(Subscription.expires_at IS NULL OR Subscription.expires_at > now())` 1차 조회 → None이면 404. **`expired` status는 항상 미반환**(과거 구독 — 신규 신청 가능 케이스).
     - sort: `started_at DESC` LIMIT 1 — 가장 최근 *유효* 구독 1건.
   - **응답 shape 변경 0건** — `SubscriptionResponse.status: Literal["active", "cancelled", "expired"]`(6.1 baseline)이 이미 `cancelled`를 wire에 노출. 클라이언트는 `status === "cancelled"` 분기로 *"해지됨 — {expires_at} 까지 이용 가능"* 표시.
   - **`subscribe` endpoint(6.1) 가드 정합** — `_subscribe_idempotent_or_replay`의 active 가드(`select(Subscription).where(Subscription.status == 'active')`)는 변경 X. cancelled-but-not-expired 사용자는 partial UNIQUE invariant상 신규 active row 작성 가능(*해지 후 재신청* 정상 케이스 — epics.md:838 *"한 사용자에 1 active subscription"* 정합).
   - **테스트 신규 ~3건** `api/tests/api/v1/test_payments.py`:
     1. `GET /subscription` cancelled-but-not-expired → 200 + `status="cancelled"` + `cancelled_at != None` + `expires_at > now()`.
     2. `GET /subscription` cancelled-AND-expired(`expires_at < now()`) → 404 + `code=payments.subscription.not_found`.
     3. `GET /subscription` active 후 cancel → cancel 호출 → 같은 endpoint 재호출 → 200 + `status="cancelled"` (1 round-trip 흐름).

   - **`SubscriptionAlreadyCancelledError(PaymentError)` 신규** — `api/app/core/exceptions.py`에 추가:
     - `status: ClassVar[int] = 409`.
     - `code: ClassVar[str] = "payments.subscription.already_cancelled"`.
     - `title: ClassVar[str] = "Subscription already cancelled"`.
     - **사유**: 이미 cancelled 상태에서 같은 endpoint 재호출 시 — *멱등 응답 200* 대신 *409 명시 분기*로 사용자 UX 명확(*"이미 해지되었습니다 — {expires_at} 까지 이용 가능"*). 클라이언트는 409 catch 후 활성 카드 갱신.
   - **service `cancel_subscription` 가드 분기**:
     - Step 1에서 `active` row 미발견 시:
       - **분기 (a)** — *cancelled-but-not-expired row 존재* → `SubscriptionAlreadyCancelledError(409)` raise.
       - **분기 (b)** — *완전 미신청 또는 expired row만 존재* → `NoActiveSubscriptionError(404)` raise(*"활성 구독이 없습니다"*).
     - 분기 결정 query: `select(Subscription).where(user_id=..., status='cancelled', expires_at > now()).limit(1)` 2차 SELECT(active miss 후 1회만).
   - **테스트 ~3건** `api/tests/services/test_payment_service.py` + `api/tests/api/v1/test_payments.py`:

     3. `cancel` 이미 cancelled-but-not-expired → `SubscriptionAlreadyCancelledError(409)` + `code=payments.subscription.already_cancelled`.

4. **AC4 — `GET /v1/payments/history` endpoint 신설 + 시계열 page + 영수증 URL 노출**
   **Given** epics.md:853 *"`GET /v1/payments/history` → `payment_logs`의 사용자 결제 이력(시점·금액·상태) 반환 + 영수증 링크 (Toss/Stripe sandbox 영수증 URL)"* + Story 6.1 `idx_payment_logs_user_id_occurred_at = (user_id, occurred_at DESC)` 시계열 인덱스 SOT + `PaymentLog.raw_payload['receipt']['url']` Toss 응답 표준 필드(`_mask_payment_payload`가 보존), **When** 사용자가 결제 이력 조회, **Then**:
   - **신규 endpoint** `GET /v1/payments/history` — `api/app/api/v1/payments.py`에 추가:
     - **Pydantic 모델 신규**:
       - `PaymentHistoryItem(BaseModel)` — wire shape 확장(`PaymentLogResponse` baseline + `receipt_url`):
         - `id: uuid.UUID`.
         - `event_type: Literal["subscribe", "cancel", "renew", "failed", "refund"]`.
         - `status: Literal["success", "failed", "pending"]`.
         - `amount_krw: int`.
         - `occurred_at: datetime`.
         - `provider: Literal["toss", "stripe"]`.
         - `receipt_url: str | None` — `raw_payload['receipt']['url']` 추출 (Toss 표준). 부재 시 None(cancel/failed event는 일반적으로 None).
       - `PaymentHistoryResponse(BaseModel)`:
         - `items: list[PaymentHistoryItem]`.
         - `next_cursor: str | None` — 다음 페이지 cursor (마지막 row의 `(occurred_at_iso8601, id_uuid)` 쌍을 `:` 구분 base64 인코딩, 부재 시 None — 마지막 페이지). cursor 형식은 안전한 정렬 invariant 보존(*`occurred_at DESC, id DESC` 복합*).
     - **endpoint 시그니처**:
       ```python
       @router.get(
           "/history",
           status_code=status.HTTP_200_OK,
           response_model=PaymentHistoryResponse,
           responses={
               200: {"description": "Payment history page (most-recent-first)"},
               400: {"description": "Invalid cursor or limit"},
           },
       )
       async def get_payment_history(
           db: DbSession,
           user: Annotated[User, Depends(current_user)],
           limit: Annotated[int, Query(ge=1, le=50)] = 20,
           cursor: Annotated[str | None, Query(max_length=200)] = None,
       ) -> PaymentHistoryResponse: ...
       ```
     - **Dependency**: `Depends(current_user)` 단독 — PIPA Art.35 *자기 결제 이력 열람*은 동의 철회와 독립.
   - **service 함수 신설** `payment_service.list_payment_history(...)`:
     - `async def list_payment_history(session: AsyncSession, *, user_id: uuid.UUID, limit: int, cursor: str | None) -> tuple[list[PaymentLog], str | None]`:
     - **Step 1 — cursor 디코딩**:
       - `cursor is None` → 1페이지(skip 0).
       - 송신 시 — `_decode_history_cursor(cursor) -> tuple[datetime, uuid.UUID]`. base64 + `:` 분리 → `(occurred_at_iso, id_str)` → `datetime.fromisoformat` + `uuid.UUID`. 형식 위반 → `PaymentHistoryCursorInvalidError(400, "payments.history.cursor.invalid")`.
     - **Step 2 — query**:
       - 1페이지: `select(PaymentLog).where(PaymentLog.user_id == user_id).order_by(PaymentLog.occurred_at.desc(), PaymentLog.id.desc()).limit(limit + 1)`.
       - cursor 페이지: 위 query에 `((PaymentLog.occurred_at, PaymentLog.id) < (cursor_occurred_at, cursor_id))` (tuple 비교 — Postgres native row-value compare) 추가.
       - 인덱스 `idx_payment_logs_user_id_occurred_at` hit (시계열 sort).
     - **Step 3 — page boundary**:
       - `len(rows) > limit` → `next_cursor = _encode_history_cursor(rows[limit-1].occurred_at, rows[limit-1].id)` + `items = rows[:limit]`.
       - `len(rows) <= limit` → `next_cursor = None` + `items = rows`.
   - **`_extract_receipt_url(raw_payload: dict[str, Any] | None) -> str | None` 헬퍼** — `payment_service.py`에 추가:
     - `raw_payload is None` → None.
     - `raw_payload.get("receipt")` dict이고 `url` key 있으면 → `str(receipt_url)` 또는 None(URL은 https://로 시작하는지 sanity 검증 — `https://`/`http://` prefix 외는 None 반환, *XSS hardening*).
     - cancel/failed event는 일반적으로 receipt 부재 → None.
   - **`_payment_log_to_history_item(log: PaymentLog) -> PaymentHistoryItem` 헬퍼** — 라우터에서 호출. `_payment_log_to_response`(6.1)와 분리 — `subscribe` endpoint 응답은 `receipt_url` 없는 baseline 유지(스코프 분리).
   - **`PaymentHistoryCursorInvalidError(PaymentError)` 신규** — `api/app/core/exceptions.py`:
     - `status: ClassVar[int] = 400`.
     - `code: ClassVar[str] = "payments.history.cursor.invalid"`.
     - `title: ClassVar[str] = "Pagination cursor invalid"`.
   - **테스트 ~6건** `api/tests/api/v1/test_payments.py` + `api/tests/services/test_payment_service.py`:
     1. `GET /history` 미신청 사용자 → 200 + `items=[]` + `next_cursor=null`.
     2. `GET /history` 1건 (subscribe) → 200 + `items` 1건 + `receipt_url` 추출(raw_payload['receipt']['url'] 보존 fixture).
     3. `GET /history` 2페이지 — limit=2 + 결제 5건 fixture → 1페이지 2건 + `next_cursor != null` → 2페이지 cursor 호출 → 2건 + cursor → 3페이지 1건 + `next_cursor=null`.
     4. `GET /history` cursor 형식 위반(`?cursor=invalid_b64`) → 400 + `code=payments.history.cursor.invalid`.
     5. `GET /history` `limit=51` → 422 (FastAPI Query `le=50` 자동 검증).
     6. `_extract_receipt_url` unit — `{"receipt": {"url": "https://..."}}` → 추출 / `{"receipt": {"url": "javascript:..."}}` → None / `{"receipt": "string"}` → None / `None` → None.

5. **AC5 — `_mask_payment_payload` `receipt` whitelist 정합 + 테스트 가드**
   **Given** Story 6.1 `_mask_payment_payload`(`api/app/adapters/toss.py:123`)는 `customerEmail`/`customerName`/`customerMobilePhone`/`secret`/`mId` 제거 + `card.number` 4자리 보존 + 그 외 모든 키(`receipt`/`easyPay`/`virtualAccount` 등)는 *기본 보존* 정합 + Story 6.1 deferred(`_mask_payment_payload` shallow — nested customer.* PII / non-string card.number)는 Story 8.4 forward, **When** 본 스토리에서 영수증 URL 추출 contract 추가, **Then**:
   - **`_mask_payment_payload` 변경 0건** — 6.1 baseline이 이미 `receipt` 키를 보존(_MASK_REMOVE_KEYS에 미포함). 본 스토리는 *코드 변경 없이* contract만 명시 + 회귀 가드 테스트 추가.

### `cancelled-but-not-expired` UX contract 결정 (Story 6.1 deferred 흡수)

Story 6.1 deferred *"`cancelled-but-not-expired` UX gap — get_subscription 404 반환"*은 본 스토리 cancel 흐름과 *직접 인접*하므로 본 스토리에서 흡수.

**채택**: `GET /subscription`이 `status.in_(['active', 'cancelled'])` AND `(expires_at IS NULL OR expires_at > now())` 1차 조회 → None이면 404. `expired` row는 항상 미반환.

**대안(검토 후 미채택)**:
- (a) `GET /subscription/active`만 active 반환 + 별 endpoint `GET /subscription/cancelled` 분리: endpoint 2건 → 클라이언트 분기 부담 + UX 모순(*"내 구독"*은 단일 개념).
- (b) `GET /subscription` 응답에 `is_active_until_expiry: bool` 별 flag: 신규 wire field → 클라이언트가 `status === "cancelled"`로 충분 분기 가능 + flag 추가는 over-engineering.

**클라이언트 분기 분배**:
- `status === "active"` + `cancelled_at === null` → *"활성 구독 — 다음 결제일 {expires_at}"* + *"해지"* 버튼 노출.
- `status === "cancelled"` + `cancelled_at !== null` + `expires_at > now()` → *"해지됨 — 이용 가능 종료일 {expires_at}"* + *"해지"* 버튼 미노출 + *"새 구독 신청"* link 노출.
- 응답 404(*expired 또는 미신청*) → *"구독이 없습니다"* + *"구독 신청"* link 노출.

### `cancel` audit `payment_logs` row 1건 INSERT 결정

**채택**: cancel 시 `payment_logs`에 `event_type="cancel"` row 1건 INSERT(`amount_krw=0` + `provider_payment_key=None` + `provider_order_id=None` + `raw_payload={"reason": "user_initiated"}`).

**사유**:
- audit-trail 정합 — 6.1 SOT *"`payment_logs`는 audit-trail 성격이라 영구 보존"* + `event_type` ENUM에 `cancel` 이미 forward-compat 등재.
- `GET /history` 흐름 단순화 — cancel도 history list에 자연 노출(*"구독 해지"* 행) → 사용자가 cancel 시점 확인 가능.
- 영수증/환불 부재 — `provider_payment_key=None` + `amount_krw=0`. webhook(6.3) 도입 후 환불 시 별 row(`event_type="refund"` + `amount_krw=환불액`) INSERT.

**대안(검토 후 미채택)**:
- (a) `subscriptions.cancelled_at`만 set + `payment_logs` INSERT X: history list에 cancel이 누락 → 사용자가 "왜 해지됐는지" 모호 + audit-trail 불완전.
- (b) `event_type="cancel"`+`status="success"` vs `status="pending"`: cancel은 *즉시 완료* 액션이므로 `success` 정합.

### `GET /history` cursor pagination 결정 — `(occurred_at, id)` 복합 row-value compare

**채택**: cursor는 `(occurred_at_iso, id_uuid_str)` 쌍을 base64로 인코딩 + Postgres native row-value compare로 안정 정렬.

**사유**:
- 같은 `occurred_at`(ms 정밀도) row가 다수일 때(*매우 드물지만*) — `id`로 tie-breaker → 사용자가 *"같은 시점 결제가 두 번 보이거나 누락"* 안 함.
- Postgres `(a, b) < (c, d)`는 표준 SQL row-value compare — *`a < c OR (a = c AND b < d)`* 동등(SQLAlchemy `tuple_(...)` 사용).
- 인덱스 `idx_payment_logs_user_id_occurred_at = (user_id, occurred_at DESC)` hit — `id` tie-breaker는 in-memory sort(드문 케이스라 비용 무시).

**대안(검토 후 미채택)**:
- (a) `offset`/`limit` 단순 pagination: 새 row가 페이지 사이에 INSERT되면 *같은 row가 두 번 보이거나 누락* — 결제 이력은 시점 invariant 보존 의무.
- (b) `occurred_at`만 cursor: tie-breaker 부재 → 같은 시점 row 다수 시 race.
- (c) opaque cursor(JWT 등): 단순 cursor에 over-engineering. base64+`:` 분리만으로 충분(클라이언트가 cursor를 파싱 X — opaque로 취급).

**page size 결정**: `limit=20` default + `le=50` cap. 모바일은 *최근 20건* 단일 page만 노출(무한 스크롤 미스코프 — Story 8.x polish forward). Web은 *"더 보기"* 버튼으로 명시 페이지 전환.

### `_extract_receipt_url` XSS hardening 결정

**채택**: `raw_payload['receipt']['url']`이 str이고 `https://` 또는 `http://` prefix면 반환, 그 외 None.

**사유**:
- Toss 응답 `receipt.url`은 *"https://dashboard.tosspayments.com/receipt/..."* 표준 — prefix 검증으로 hardening.
- `javascript:` / `data:` URL이 raw_payload에 무리한 케이스에서 들어왔을 때 클라이언트 *"영수증"* 버튼 click → XSS 방어.
- 클라이언트(Mobile `Linking.openURL` + Web `<a target="_blank" rel="noopener noreferrer">`)도 *2차 hardening* — 신뢰는 *서버 1차 검증* + *클라이언트 2차 hardening* 양 layer.

**대안(검토 후 미채택)**:
- (a) prefix 검증 X — 클라이언트가 모두 책임: 신뢰 경계 단일점 → 서버 응답이 *"믿을 만"*해야 함.
- (b) `urllib.parse.urlparse` 정밀 검증: prefix 검증으로 충분(`https://` prefix 위반 = parse 실패 / scheme 미일치) + `urlparse` 추가 cost.
