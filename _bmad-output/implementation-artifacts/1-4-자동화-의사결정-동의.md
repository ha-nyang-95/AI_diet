## Story

As a **엔드유저**,
I want **PIPA 2026.03.15 자동화 의사결정 동의를 별도로 수행하기를**,
So that **미동의 시 시스템이 LangGraph 분석 호출을 차단하고 안내 화면을 노출함으로써 LLM 비용·법적 의무를 모두 보호받는다.**

부수적으로 본 스토리는 Epic 3(분석) 진입 *전*에 박아야 하는 두 번째 동의 게이트(`require_automated_decision_consent`)와 동의 철회 흐름까지 정립한다 — Story 1.3이 박은 `consents` 테이블에 자동화 의사결정 컬럼 3개를 0004 마이그레이션으로 추가하고, `app/domain/legal_documents.py` SOT에 `automated_decision` 타입 신규 도입(한·영 5섹션 풀텍스트). Story 1.6이 본 스토리의 onboarding 화면을 chain(`privacy → automated-decision → tutorial → profile`)에 끼우고, Epic 3 분석 라우터(`/v1/analysis/stream`)가 본 스토리의 `require_automated_decision_consent` Depends를 명시 wire한다. 함께 Story 1.3 deferred W13(`X-Consent-Latest-Version` 응답 헤더)을 본 스토리 409 패턴 도입 시점에 일괄 흡수한다.

1. **AC1 — 자동화 의사결정 동의 화면 5섹션 노출 + 별도 체크 박스**
   **Given** 신규 또는 기존 사용자(Story 1.3 4종 동의 통과 직후), **When** 모바일이 `/(auth)/onboarding/automated-decision` 진입, **Then** 다음 5섹션이 단일 화면(`ScrollView` 내) 풀텍스트로 노출되며, *"위 자동화된 의사결정 처리에 별도 동의합니다 (PIPA 2026.03.15)"* 별도 체크 박스 + *동의 완료* 버튼 노출:
     1. **처리 항목** — 체중·신장·알레르기·식사 기록·건강 목표(NFR-O2 마스킹된 LLM 입력)
     2. **이용 목적** — 자동화된 영양·식단 분석(LangGraph 6노드 + Self-RAG)·`fit_score` 산출·인용 피드백 생성
     3. **보관 기간** — 분석 결과(`meal_analyses`) 및 트레이스(LangSmith)는 회원 탈퇴 시 즉시 파기. 동의 시점·철회 시점 기록은 PIPA 외 관련법 보존 의무 만료 시까지 격리 보관
     4. **제3자 제공** — 없음. *국외 이전*: OpenAI(미국) / Anthropic(미국) / LangSmith(미국)
     5. **LLM 외부 처리** — OpenAI GPT-4o-mini / Anthropic Claude (`adapters/{openai,anthropic}.py`) 호출, NFR-O2 마스킹 processor가 PII(이메일·체중·식단 raw_text·알레르기) 송신 전 마스킹 처리. `LANGCHAIN_TRACING_V2` env가 활성화된 경우 마스킹된 trace가 LangSmith에 송신
   - 단일 "전체 동의" 위장 금지(PIPA 22조 별도 동의 의무 정합 — Story 1.3 AC1 패턴 동일).
   - 본 화면은 *별도 체크 박스 1개*(automated_decision)만 노출 — Story 1.3의 4 체크와는 별 화면. 화면 진입 가드: `consentStatus.basic_consents_complete=false`이면 즉시 `(auth)/onboarding/disclaimer`로 redirect(`?next=` 보존, Story 1.6에서 처리).
   - **본 스토리에서 라우트는 mobile primary** + **Web minimal UI 동시 도입**(영업 데모 lock-in 회피, Story 1.3 AC14 패턴 정합).

2. **AC2 — 미동의 사용자 분석 라우터 차단 + RFC 7807 403 응답**
   **Given** 자동화 의사결정 미동의 사용자, **When** 분석 라우터(`/v1/analysis/stream` 등 — Epic 3 도입) 호출, **Then** `Depends(require_automated_decision_consent)` 게이트가 LangGraph·LLM 호출 *전*에 `AutomatedDecisionConsentMissingError`(403) raise → 글로벌 핸들러가 RFC 7807 + `application/problem+json` 응답:
     ```json
     {
       "type": "about:blank",
       "title": "Automated decision consent required",
       "status": 403,
       "code": "consent.automated_decision.missing",
       "detail": "Automated decision consent is required to invoke AI analysis. Please grant the consent first.",
       "instance": "/v1/analysis/stream"
     }
     ```
   - **본 스토리에서 `/v1/analysis/stream` 라우터 자체는 도입하지 않음** — Epic 3 Story 3.x 책임. 본 스토리는 (a) `app/api/deps.py:require_automated_decision_consent` 함수 정의 + (b) 단위 테스트로 동의 여부에 따른 raise 검증만 수행. wire는 Epic 3 시점에 명시 적용 — `analysis.py` 라우터의 모든 SSE/분석 endpoint에 `Depends(require_automated_decision_consent)` 추가. Story 1.3 `require_basic_consents` 패턴 정합(함수 정의 + 단위 테스트, 라우터 wire는 사용처 스토리 책임).
   - 분석 라우터 wire 시점 비용 회귀 차단 SOP는 Story 8 polish 또는 Epic 3 retrospective.

3. **AC3 — 동의 시 `consents` row에 컬럼 3개 기록 + UPSERT 패턴**
   **Given** 모바일/Web에서 사용자가 *동의 완료* 탭, **When** `POST /v1/users/me/consents/automated-decision`(body: `{automated_decision_version: string}` — `GET /v1/legal/automated-decision` 응답의 `version` 필드와 동일), **Then**:
     - 백엔드가 `consents` 테이블 row를 user 기준으로 갱신:
       - `automated_decision_consent_at = now()` set
       - `automated_decision_version = body.automated_decision_version` set
       - `automated_decision_revoked_at = NULL`로 reset(이전에 철회한 사용자가 재동의하는 케이스 — `automated_decision_consent_at`만 새 시각으로, `automated_decision_revoked_at`는 NULL로 명시 clear)
       - `ip_address`, `user_agent` 갱신 — Story 1.3 AC3 패턴 정합
       - `updated_at = now()` 갱신(`ON CONFLICT DO UPDATE.set_`에 명시 — Story 1.3 P2 패턴)
     - **Story 1.3 ` consents` row가 없는 사용자**(이론상 불가 — 본 화면 진입 가드가 basic 4종 통과 강제)도 안전하게 동작 — `INSERT ... ON CONFLICT DO UPDATE`로 row 부재 시 INSERT(basic 4 컬럼은 NULL 상태). 단 *기대 흐름*은 row 이미 존재이며, row 부재 케이스에 대한 명시 테스트는 본 스토리 OUT(가드가 차단).
     - 응답 status **200** + body `ConsentStatusResponse`(Story 1.3 응답 모델 + 본 스토리 신규 4 필드 — AC4) — `automated_decision_consent_complete: true`. 신규/갱신 구분 status 코드(201 vs 200) 도입은 본 스토리 *철회→재동의* 의미상 모호 → **항상 200** 채택(Story 1.3는 `xmax = 0` returning으로 신규 vs 갱신 구분했으나 본 endpoint는 부분 update로 의미가 다름).
     - 클라이언트가 보낸 `automated_decision_version`이 backend 현재 SOT 버전(`CURRENT_VERSIONS["automated-decision"]`)과 다르면 **409 Conflict** + `code=consent.automated_decision.version_mismatch` (Story 1.3의 `consent.version_mismatch`는 4종 basic 전용으로 유지 — 게이트별 명확한 코드 분리). **응답 헤더에 `X-Consent-Latest-Version: <SOT 버전>` 동시 첨부**(Story 1.3 deferred W13 흡수 — 클라이언트가 추가 round trip 없이 latest version 즉시 인식). Story 1.3 `ConsentVersionMismatchError`도 같은 헤더 패턴 도입(아래 AC11).
     - 누락 필드(예: body 미전송) → **400** RFC 7807 + `code=validation.error` (`main.py`의 `validation_exception_handler` 매핑 — Story 1.2/1.3 정합).

4. **AC4 — `ConsentStatusResponse` 응답 모델 확장**
   **Given** 모든 `consents` 라우터(`POST /v1/users/me/consents`, `GET /v1/users/me/consents`, `POST /v1/users/me/consents/automated-decision`, `DELETE /v1/users/me/consents/automated-decision`)의 응답, **When** 클라이언트가 응답을 파싱, **Then** 다음 4 필드가 추가로 노출(snake_case 양방향, Story 1.3 ConsentStatusResponse 정합):
     - `automated_decision_consent_at: datetime | null`
     - `automated_decision_revoked_at: datetime | null`
     - `automated_decision_version: str | null`
     - `automated_decision_consent_complete: bool` — 4 조건 동시 만족: (a) `automated_decision_consent_at IS NOT NULL`, (b) `automated_decision_revoked_at IS NULL`, (c) `automated_decision_version == CURRENT_VERSIONS["automated-decision"]`, (d) Story 1.3 `basic_consents_complete=true`(*) — 단 (d)는 본 응답 필드의 *기본 동의 선행* 의미 강화용이며 라우터 게이트(`require_automated_decision_consent`)는 (a)·(b)·(c)만 검증한다. (d)를 응답 필드에 합쳐야 클라이언트가 단일 boolean으로 분기 가능.
   - **별 도메인 결정**: (d) 포함 vs 분리 — 본 스토리는 *포함* 채택. 사유: 클라이언트는 보통 *기본 + 자동화 결정 모두 통과* 여부만 관심. 분리 시 `basic_consents_complete && automated_decision_consent_complete`를 매번 결합해야 함 → 응답 측에서 통합. *(검증 로직 분리* — `has_all_basic_consents` / `has_automated_decision_consent` 두 함수는 분리 유지 → 게이트별 책임 명확.)*
   - 기존 4 필드(`disclaimer_acknowledged_at` 등) 응답 위치·이름 변경 X — Story 1.3 정합 + OpenAPI 클라이언트 회귀 차단.

5. **AC5 — `DELETE /v1/users/me/consents/automated-decision` 동의 철회 endpoint**
   **Given** 자동화 의사결정 동의를 이미 부여한 사용자, **When** 설정에서 *철회* 액션 → `DELETE /v1/users/me/consents/automated-decision` 호출, **Then**:
     - 백엔드가 `consents` row 갱신: `automated_decision_revoked_at = now()` set. `automated_decision_consent_at`은 보존(과거 동의 사실은 PIPA 외 관련법 감사 흔적).
     - 응답 status **200** + body `ConsentStatusResponse`(`automated_decision_consent_complete: false`).
     - row가 없거나 `automated_decision_consent_at IS NULL`(미동의 상태) → **404 Not Found** + `code=consent.automated_decision.not_granted` (이미 미동의 상태에서 추가 철회 의미 없음 — idempotency 정합 X 케이스). RFC 9110 §15.5.5 정합. 클라이언트는 GET으로 상태 재확인 후 분기.
     - 응답 후 즉시 `Depends(require_automated_decision_consent)` 게이트가 차단 작동 — Epic 3 wire 후 다음 분석 호출에서 RFC 7807 403 노출(AC2 정합). 본 스토리는 *DELETE 동작 + 단위 테스트*만, 분석 라우터 wire는 Epic 3.

[Source: architecture.md#Authentication & Security (line 311), architecture.md#Format Patterns (line 462-471), architecture.md#Cross-Cutting Concerns (line 890), Story 1.3 Dev Notes "동의 게이트 분리"]

**동의 게이트 3종 분리 (절대 통합 금지) — 본 스토리에서 3번째 박힘**

- **인증 게이트** (`current_user`, Story 1.2): JWT + DB lookup. 모든 보호 라우트.
- **기본 동의 게이트** (`require_basic_consents`, Story 1.3): 4종 동의 + SOT 버전 일치. 핵심 사용자 데이터 작성·조회 라우트(Story 1.5+).
- **자동화 의사결정 게이트** (`require_automated_decision_consent`, **본 스토리**): PIPA 2026.03.15 별도 동의 + 미철회 + SOT 버전 일치. *분석 호출* 라우터(`/v1/analysis/stream` — Epic 3 Story 3.x wire).
- **이유**:
  - `/v1/users/me`(자기 정보) — 인증만 충분, 동의 무관.
  - `/v1/users/me/profile`(건강 프로필) / `/v1/meals/*`(식단) — 인증 + 기본 동의 필요(민감정보 처리 의무).
  - `/v1/analysis/stream`(LLM 분석) — 인증 + 기본 동의 + 자동화 의사결정 동의(PIPA 2026.03.15). 미동의 시 LangGraph 호출 *전* 차단 → LLM 비용 0 보호.
- **wire 정책**: 본 스토리는 *함수 정의 + 단위 테스트*만. wire는 사용처 스토리 책임. Story 1.3 `require_basic_consents` 패턴 정합.

**자동화 의사결정 SOT 분리 (`automated-decision` 타입 신규)**

- Story 1.3은 `disclaimer`/`terms`/`privacy` 3종 + `sensitive_personal_info`(privacy 텍스트 공유). 본 스토리는 `automated-decision`을 **새 타입**으로 분리 도입.
- 분리 이유: PIPA 2026.03.15 시행 자동화 의사결정 동의는 별도 5섹션 명세 의무(처리 항목·이용 목적·보관 기간·제3자 제공·LLM 외부 처리) — privacy 텍스트와 *공유 단락*이 아닌 *별도 문서*. 향후 클라이언트 외주 인수 시 AD 텍스트만 갱신할 가능성 + 입법예고(R9) 추적 시 별 buffer.
- `LegalDocType` Literal 확장: `Literal["disclaimer", "terms", "privacy", "automated-decision"]` (hyphen — URL/SDK method 단순화). `CURRENT_VERSIONS` 매핑 키도 hyphen 통일(*"automated-decision"*).
- 본 스토리 시점에는 `_VERSION_KO` 공유로 시작 → 향후 AD 텍스트만 갱신될 경우 `_VERSION_AUTOMATED_DECISION_KO` 별 const로 분기 + `CURRENT_VERSIONS["automated-decision"]` 독립 bump. 본 분기 결정은 주석으로 모듈에 명시.

**`consents` 테이블 컬럼 per 동의 (event-style 미채택) — 본 스토리에서 3컬럼 추가**

- Story 1.3 결정 정합 — Option A(컬럼 per 동의) 채택. 본 스토리는 0004 마이그레이션으로 `automated_decision_consent_at`, `automated_decision_revoked_at`, `automated_decision_version` 3컬럼 추가.
- **철회 의미 명시**: `automated_decision_revoked_at` 컬럼은 *철회 시점* 기록 — 사용자가 다시 동의 시 NULL로 reset(과거 철회 사실 자체는 `consents` 테이블 기록보다 `audit_logs`(Story 7.3) 자동 기록에 의존). `automated_decision_consent_at`은 *최신 동의 시점*만 보존 — 과거 동의 history는 `audit_logs`에서.
- 단일 row 유지(Story 1.3 정합 — 1 user 1 row). UPSERT는 `pg_insert(...).on_conflict_do_update(set_={...automated_decision_*})` 단일 statement.

**RFC 7807 응답 헤더 `X-Consent-Latest-Version` 도입 (Story 1.3 deferred W13 흡수)**

[Source: deferred-work.md `W13`, RFC 9110 §5.6.1 list-based header]

- 409 Conflict 응답에 `X-Consent-Latest-Version` 헤더 첨부 — 클라이언트가 추가 round trip(GET /v1/legal/{type})없이 latest version 즉시 인식.
- 헤더 인코딩: `<doc_type>=<version>` 단일 entry, 다중 entries 시 `; ` 구분(예: `disclaimer=ko-2026-04-28; automated-decision=ko-2026-04-28`). RFC 9110 §5.6.1 list-based 헤더 패턴 정합.
- 본 스토리는 *첫 mismatch entry만* 헤더에 인코딩 — UX 노이즈 회피 + 클라이언트 단순성. 4 mismatch 동시 발생 시도 첫 mismatch 한 entry로 응답 → 클라이언트가 GET으로 fresh 텍스트 fetch 후 모두 갱신.
- 패턴 적용 대상: `ConsentVersionMismatchError`(Story 1.3) + `AutomatedDecisionConsentVersionMismatchError`(본 스토리) 두 예외 동시. 다른 도메인 예외에 일반화는 본 스토리 OUT — 동의 영역 전용.
- 클라이언트는 `body.code` 1차 신호 + 헤더는 *옵션 hint*로만 사용. 헤더 검증 의무는 spec 비요구.

### Library / Framework 요구사항

[Source: Story 1.3 Dev Notes "Library 요구사항"]

| 영역 | 라이브러리 | 정확한 사유 |
|------|-----------|-----------|
| UPSERT (POST grant) | `sqlalchemy.dialects.postgresql.insert` | 단일 statement UPSERT — race condition 차단(Story 1.3 정합). re-grant 시 `automated_decision_revoked_at=None`로 reset |
| UPDATE (DELETE revoke) | `sqlalchemy.update(Consent).where(...).values(...)` | row 존재 보장된 상태(404 게이트가 사전 raise)에서 단일 UPDATE statement. SELECT-then-UPDATE race는 본 스토리 시점 multi-replica X(Story 1.1 deferred 정합) — 단일 replica에서 부분 update 안전 |
| 모바일 confirm dialog | RN `Alert.alert(title, message, buttons)` | 본 스토리 1회 사용 — 추가 라이브러리 도입 회피. 풀 confirm UX는 Story 5.x(회원 탈퇴) 시점 결정 |
| 모바일 form | `useState<boolean>` + 기본 `Pressable` | Story 1.3 패턴 정합 — `react-hook-form`은 Story 1.5(건강 프로필)에서 본격 도입 |

### Naming 컨벤션

[Source: architecture.md#Naming Patterns (line 386-431), Story 1.3 Dev Notes "명명 규칙"]

- DB: `consents` 테이블, `automated_decision_consent_at` / `automated_decision_revoked_at` / `automated_decision_version` snake_case.
- API URL: `POST /v1/users/me/consents/automated-decision`, `DELETE /v1/users/me/consents/automated-decision` (kebab-case path segment), `GET /v1/legal/automated-decision`. 단수 group 유지(Story 1.3 패턴 정합).
- JSON wire: snake_case 양방향. `automated_decision_consent_at`, `automated_decision_revoked_at`, `automated_decision_version`, `automated_decision_consent_complete`. 요청 body는 `{automated_decision_version: "..."}` 단일 필드.
- 파일명: `0004_consents_automated_decision.py` (alembic), `automated-decision.tsx` (mobile/web Expo Router/Next.js — kebab-case).
- React Native 파일: `mobile/app/(auth)/onboarding/automated-decision.tsx`, `mobile/app/(tabs)/settings/automated-decision.tsx`.
- 도메인 예외 클래스: `AutomatedDecisionConsentMissingError` / `NotGrantedError` / `VersionMismatchError` PascalCase.
- 코드 카탈로그: `consent.automated_decision.missing` / `consent.automated_decision.not_granted` / `consent.automated_decision.version_mismatch` lowercase + `.` 구분(Story 1.3 정합 — `consent.basic.missing` / `consent.version_mismatch` 패턴).

### 보안·민감정보 마스킹 (NFR-S5)

[Source: prd.md#NFR-S5, NFR-O2, Story 1.3 Dev Notes]

- `consents` POST/DELETE 라우터는 user_id만 로깅 — `event=consents.automated_decision.granted user_id=<uuid>` / `event=consents.automated_decision.revoked user_id=<uuid>`. 이메일·동의 정보 raw 노출 금지(Story 1.3 정합).
- `ip_address`/`user_agent`는 DB 저장 — 로그·Sentry breadcrumb 노출 X(Story 1.3 W2 deferred 정합).
- 본 스토리는 LLM 호출을 *직접 호출하지 않음* → NFR-O2 마스킹 processor는 본 스토리 시점에 활성화 X(Epic 3 Story 3.x). 단 AD 동의서 텍스트(AC1 섹션 5)에 LLM 외부 처리 + NFR-O2 마스킹 정책을 *명시*해 사용자 고지 의무 충족.

### Anti-pattern 카탈로그 (본 스토리 영역)

[Source: Story 1.3 Anti-pattern 카탈로그, architecture.md#Anti-pattern (line 557-565)]

- ❌ 자동화 의사결정 동의를 기본 4종 동의 화면 안에 *5번째 체크박스*로 통합 — PIPA 2026.03.15 별도 동의 의무 위반(별 화면 강제).
- ❌ 자동화 의사결정 동의 텍스트를 privacy.body 단락에 합치기 — `automated-decision` 별 SOT 분리 필수(R9 입법예고 추적 + 외주 텍스트 갱신 격리).
- ❌ `require_automated_decision_consent`를 `/v1/users/me` 또는 `/v1/users/me/profile`에 wire — 본 게이트는 *분석 호출* 전용. 사용자 자기 정보 조회 / 건강 프로필 작성은 LLM 호출 무관(미동의 사용자도 가능).
- ❌ `require_automated_decision_consent`와 `require_basic_consents`를 *통합 wrapper*로 합치기 — 게이트별 책임 명확화 위해 분리 유지(분석 라우터는 둘 다 wire하되 별 Depends).
- ❌ `automated_decision_revoked_at = now()` set 시 `automated_decision_consent_at` 함께 NULL로 reset — 과거 동의 사실은 보존(PIPA 외 관련법 감사 흔적).
- ❌ `DELETE /automated-decision`에서 row 없음 / consent_at NULL 케이스를 200 OK + idempotent 반환 — 사용자가 *철회 의도*인데 *미동의 상태*는 의미 모호 → 404 + `code=consent.automated_decision.not_granted`로 명시 분기.
- ❌ `pg_insert(...).values(automated_decision_consent_at=now)` 시 INSERT 분기에서 basic 4 컬럼을 *함께* set — 기본 4 동의가 없는 사용자가 본 endpoint를 호출하는 것 자체가 *예외 경로*(가드가 차단)이며, INSERT 시도 시 basic 4를 함께 채우면 *동의하지 않은 항목까지 false 통과* 위험. 본 스토리는 INSERT 분기에서도 basic 4 컬럼 미지정(NULL 유지) → `has_all_basic_consents=false` 반환으로 클라이언트가 onboarding 흐름 재진입 강제.
- ❌ `ConsentStatusResponse`에 신규 4 필드 추가 시 기존 필드 위치 변경 또는 이름 변경 — Story 1.3 OpenAPI 클라이언트 회귀 + Story 1.5+ wire 시점 회귀 위험. *추가만, 변경 X*.
- ❌ `X-Consent-Latest-Version` 헤더를 *모든 4xx 응답*에 첨부 — 본 헤더는 409 version mismatch 전용 hint. 일반화 금지(클라이언트 파싱 부담 + 정보 누설).
- ❌ `legal.py`의 path Literal에 `"automated-decision"` 추가 시 SOT dict 키와 *비대칭*(예: SOT는 `"automated_decision"` underscore vs URL은 `"automated-decision"` hyphen) — Literal 값과 dict 키 모두 hyphen 통일(AC13 결정).

### Previous Story Intelligence (Story 1.3 학습)

[Source: `_bmad-output/implementation-artifacts/1-3-약관-개인정보-동의.md` Dev Agent Record + Review Findings + `deferred-work.md`]

**적용 패턴(반드시 재사용):**

- **UPSERT race 방지**: `pg_insert(...).on_conflict_do_update(...)` 단일 statement(Story 1.3 P11/AC9 패턴). 본 스토리 `POST /automated-decision`에서 동일 사용.
- **`updated_at` 명시 갱신**: `ON CONFLICT DO UPDATE.set_`에 `updated_at=func.now()` 명시(Story 1.3 P2 patch 정합). DB `onupdate=text("now()")` 이벤트는 PG 직접 DML에서 발화 X.
- **응답 모델 snake_case 양방향**: Story 1.3 P20 패턴(`alias_generator` 미사용). `ConsentStatusResponse` 확장 시 동일 정책.
- **server-only helper**: Web `getServerSideConsents` / `fetchLegalDocument` Story 1.3 P3 patch 정합 — `bn_access`만 선별 forward + `import "server-only"` 첫 줄.
- **테스트 fixture 패턴**: `_truncate_user_tables` *before yield* (Story 1.2 T1 patch). `consents` 테이블 truncate 이미 명시(Story 1.3 정합).
- **assertion 정확 매칭**: `code == "consent.automated_decision.missing"` 등 구체 코드 단언 — Story 1.2 T11 patch 정합.
- **TanStack Query authFetch 적용**: 모바일 mutation은 `authFetch` 사용(Story 1.3 정합). 401 → refresh 1회 → 재시도 자동.
- **double-tap 방지**: `inflightRef` + `submitting` state 결합(Story 1.3 P10 패턴 정합). 본 스토리 mobile/Web 동의 완료 버튼 모두 적용.
- **typed error class**: `ConsentSubmitError`(Story 1.3 P11 patch) 패턴 — 본 스토리는 신규 `AutomatedDecisionMutationError` 또는 `ConsentSubmitError` 재사용으로 분기 메시지 표시.
