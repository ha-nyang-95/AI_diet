## Story

As a 엔드유저,
I want 주간 리포트에서 *"다음 주 권장 매크로 조정"* 인사이트 카드를 한국 1차 출처 인용과 함께 보고, 매크로/칼로리 목표를 조정하면 미기록 nudge 메시지와 다음 식단 분석의 권장 표시가 자동 반영되기를,
So that 분석 → 인사이트 → 행동 → nudge·권장 자동 반영의 사이클이 손이 떨어지는 곳에서 닫힌다.

   - **단위 테스트** ~14건: protein_deficit (목표 우선순위 ①/②/③) 3 + calorie_diff (편차 25%/10-25%/<10%) 3 + macro_ratio_drift (비율 부분 set 가드 + 편차 분기) 3 + allergen_exposure (≥3건/2건/0건/다중 알레르기) 4 + 카드 우선순위 cap 1.

### 결정 노트

#### D1 — `users.macro_goal`은 JSONB vs 별도 컬럼 5건

- **선택 (JSONB)**: Pydantic SOT 1지점 + partial update merge SQL 직관(`COALESCE || patch`) + 미래 매크로 목표 필드 추가(예: `fiber_target_g`/`sodium_max_mg`) 시 마이그레이션 0건. DB CHECK 제약으로 shape 가드(double gate). LLM cache key 직렬화 시점에 `model_dump(exclude_none=True)` SOT.
- **Reject (별도 5 컬럼)**: 5 컬럼은 partial set 분기를 SQL UPDATE에서 풀어쓰기 복잡(`UPDATE ... SET col1 = COALESCE(:patch_col1, col1), ...`). 미래 필드 추가마다 alembic 1건. 본 스토리 5 필드는 하드 cap 아님 — 1년 내 7-10 필드 가능 → JSONB 정합.

#### D2 — `daily_calorie_target_kcal` SOT vs `weekly_calorie_target` (주간 SOT)

- **선택 (daily)**: nudge body의 챌린지 카피는 *하루* 단위(*"오늘 저녁"*). 인사이트 카드의 calorie_diff도 일일 평균 vs 일일 목표가 직관. 주간 SOT는 1.5만 kcal 같은 큰 숫자라 사용자 인지 부담.
- **Reject (weekly)**: 주간 합산은 *결과 표시* 영역(차트), *목표 SOT*는 일일 단위가 표준(KDRIs/대비학회 모두 일일 권장).

#### D3 — Insight 엔진 LLM vs 결정성 룰

- **선택 (결정성 룰)**: 4 카드 종류는 *수치 비교 룰*(평균 vs 목표 vs 임계값) — LLM 호출 불필요. 비용 0 + latency 0 + 결정성(cache 정합) + 단위 테스트 가능. epics 예시(*"단백질 평균 55g/일 — 목표 80g 미달"*)도 *데이터 + 임계값* 패턴.
- **Reject (LLM)**: LLM 인사이트는 *creative writing* 영역(예: *"이번 주는 회식이 많으셨군요. 주말에 채소 위주로 균형 회복해보세요"*). Story 4.3 baseline 위에서 LLM 인사이트 추가는 *별 스토리* 가치(7-10 시간 추가 + 비용 + 평가 데이터셋 필요). 본 스토리 scope 외 — Story 8.4 polish 또는 신규 별 스토리 forward.

#### D4 — Profile cache invalidate (`cache:user:{user_id}:profile`)

- **현 상태 (cache 미구현)**: architecture.md:296에 키 패턴 박혀있으나 *Redis cache 레이어 미구현* — `current_user` Dependency가 매번 DB SELECT(N+1 회피는 selectinload). 본 스토리는 cache invalidate 코드 추가 0건 + forward-compat 코멘트만 추가. Story 8.4 polish에서 Redis profile cache wire 시점에 macro_goal patch 핸들러에 `redis.delete(f"cache:user:{user_id}:profile")` 1줄 추가하는 forward-hook.
- **LLM cache invalidate**: `_build_profile_hash` 확장(AC8)으로 *자동* — macro_goal 변경 → sha256 다른 값 → 다음 분석 cache miss → 새 LLM 호출. 본 스토리 baseline에서 *분석 권장 표시 자동 반영*은 이 메커니즘으로 충족.

#### D5 — 인사이트 카드 우선순위 cap = 4

- **선택 (4 cap)**: 주간 리포트 카드 영역 가독성 + 사용자 행동 의사결정 부담 최소화. 4 카드는 한 눈에 파악 가능 + cta 1개 충분(매크로/칼로리 조정 단일 행동).
- **Reject (cap 미설정)**: 알레르기 22종 노출 ≥3건 사용자(드물지만 가능)면 카드 22+개 → cognitive overload + 리포트 페이지 길이 폭증. allergen sub-cap 2개 + 총 cap 4개 권장 절충.

#### D6 — `MacroGoalPatchRequest` `null` 송신 = key 삭제 vs key 업데이트(null로)

- **선택 (key 삭제)**: REST PATCH 표준 — `{"daily_calorie_target_kcal": null}` 송신 = 해당 키 *clear*. JSONB merge SQL 분기에서 null 값은 `macro_goal - 'key'` SQL operator로 key 자체 제거. Pydantic `model_dump(exclude_none=True)`로 patch_jsonb 빌드 후 별도 null keys list로 SQL key remove 분기.
- **Reject (key 업데이트)**: JSONB에 `{"daily_calorie_target_kcal": null}` 저장은 *값이 null인 키* — Pydantic 응답 모델은 동일하게 `None` 표현이라 wire format 동일하지만 DB row가 *명시적 null* vs *부재* 의미 구분이 모호 → 향후 필드 추가 시 schema 혼란.

#### D7 — `protein_target_g_per_meal` (per meal) vs `protein_target_g_per_day` (per day)

- **선택 (per meal)**: epics.md:773 예시 *"오늘 저녁 단백질 25g 챌린지"* + nudge body는 *한 끼* 행동 단위. 사용자 인지 직관(*"한끼당 25g"* vs *"하루 75g"* 둘 다 동일하지만 per meal이 행동 트리거).
- **Reject (per day)**: per day는 *분석 시점에 25/35/30/10 끼별 분배* 룰 추가 필요(NFR-UX5 한국 식사 패턴 SOT). 본 스토리 scope 폭증 + nudge body의 *오늘 저녁* 카피 정합 X.
