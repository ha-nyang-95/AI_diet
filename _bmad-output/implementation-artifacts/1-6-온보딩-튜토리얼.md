## Story

As a **엔드유저**,
I want **첫 사용 시 디스클레이머 → 약관·개인정보 동의 → 사용법 안내(2-3 화면) → 건강 프로필 입력의 순차 흐름을 통과하기를**,
So that **첫 사용 시 멈춤 없이 핵심 기능에 진입할 수 있다.**

부수적으로 본 스토리는 (a) Story 1.3·1.4·1.5에서 박힌 `/(auth)/onboarding/{disclaimer → terms → privacy → automated-decision → profile}` chain의 *중간*에 신규 `tutorial.tsx`(사용법 2-3 화면)를 끼워 PRD FR28의 6단계 순차 흐름을 완성하고, (b) `users.onboarded_at` 컬럼(Story 1.2 baseline)을 *프로필 첫 입력 시 1회 set* 멱등 패턴으로 박아 향후 *기존 사용자 재로그인 스킵* 시그널 + audit 흔적의 단일 출처를 만든다. (c) Mobile `(tabs)/_layout.tsx`의 4번째 가드 redirect target을 `/profile` → `/tutorial`로 변경해 tutorial이 자체 가드(AsyncStorage `tutorial-seen` flag)로 first-time vs resume 분기. (d) `UserPublic`(Story 1.2 OAuth 응답 모델)에 `onboarded_at` 노출을 추가해 모바일이 signIn 직후부터 *기존 사용자* 여부 판별. (e) Web 측 tutorial은 OUT — architecture line 985 *"FR28 | mobile `(auth)/onboarding/tutorial.tsx`"* 명시 정합(mobile-only, Web onboarding은 minimal UI 유지). 본 스토리 종료 시점에 Epic 1 **`기반 인프라 + 인증 + 온보딩·동의`** 모든 6 스토리(1.1-1.6) 완료 → Epic 2(식단 입력)로 전환할 토대 마련.

3. **AC3 — Mobile `(tabs)/_layout.tsx` 4번째 가드 redirect target 변경 + tutorial → profile chain 박기**

5. **AC5 — 백엔드 `POST /v1/users/me/profile`에 `onboarded_at = COALESCE(onboarded_at, now())` 추가**
   **Given** 본 스토리, **When** 사용자가 `POST /v1/users/me/profile`을 호출, **Then**:
     - `update(User)` SQL의 `values(...)`에 `onboarded_at=func.coalesce(User.onboarded_at, func.now())` 추가 — 기존 컬럼이 NULL이면 `now()`로 set, 이미 set이면 그대로 유지(idempotent). PRD AC *"프로필 입력 완료 시 (tabs)/meals 라우팅 + users.onboarded_at 기록"* 정합.
     - 재입력(이미 입력된 사용자가 다시 POST — Story 5.1 prefill 흐름)에서 `onboarded_at`은 *변경되지 않음* — *최초 온보딩 완료 시점* 의미 보존(audit 흔적). `profile_completed_at`은 매번 갱신(*최신 프로필 기록 시점*) — 두 컬럼 의미 분리 유지.
     - SQL 단일 statement(race-free):
       ```python
       update(User)
       .where(User.id == user.id)
       .values(
           age=body.age,
           weight_kg=body.weight_kg,
           height_cm=body.height_cm,
           activity_level=body.activity_level,
           health_goal=body.health_goal,
           allergies=body.allergies,
           profile_completed_at=func.now(),
           onboarded_at=func.coalesce(User.onboarded_at, func.now()),
           updated_at=func.now(),
       )
       .returning(User)
       .execution_options(populate_existing=True)
       ```
     - `HealthProfileResponse`에 `onboarded_at` 추가 X(*프로필 데이터*가 아닌 *온보딩 흔적* — `UserMeResponse`/`UserPublic` 책임. Story 5.1 PATCH 응답에서도 동일 분리 유지). `_build_profile_response` 변경 X.
     - 로깅: `event=users.profile.updated user_id=<uuid> onboarded_first_time=<bool>` — `onboarded_first_time`은 update *이전* `User.onboarded_at IS NULL`로 판별. PIPA 민감정보 raw 노출 차단(Story 1.5 NFR-S5 정합).

   - `mobile/features/onboarding/tutorialState.ts`에 `clearOnboardingState()` helper export 추가:

**Tutorial 화면은 *정보 제공*만 — 데이터 수집 X + 백엔드 endpoint 도입 X**

**`users.onboarded_at` vs `users.profile_completed_at` — 분리 유지(Story 1.5 결정 정합)**

[Source: 1-5-건강-프로필-입력.md Architecture Decisions (line 512-517) + Detected Conflicts (line 744)]

- `profile_completed_at` (Story 1.5): *프로필 입력 시점*. 매 POST 시 갱신.
- `onboarded_at` (본 스토리): *최초 온보딩 흐름 통과 시점*. 첫 POST 시만 set(`COALESCE`로 멱등). 이후 재 POST에도 변경 X.
- 두 컬럼 의미 분리: profile_completed_at은 *기술 게이트*(Story 1.4·1.5 4번째 가드 입력), onboarded_at은 *사용자 경험 단계 통과*(Story 8 audit/welcome 메시지 입력).
- Story 5.1 *프로필 수정* 도입 시 — `PATCH /v1/users/me`(또는 `POST /me/profile`) 호출이 `profile_completed_at`만 갱신, `onboarded_at`은 `COALESCE`로 그대로 유지(본 스토리에서 박은 패턴 그대로 재사용).

**`onboarded_at` 멱등 set은 `COALESCE` SQL이 가장 단순 — Python read-then-write 회피**

- 대안: Python에서 `if user.onboarded_at is None: ...values(onboarded_at=now())` 분기 — race condition(다른 device 동시 POST) + 추가 SELECT 비용 + 코드 분기 복잡. 회피.
- `func.coalesce(User.onboarded_at, func.now())` SQL 표현식 — 단일 UPDATE statement 내에서 *기존 값이 NULL이면 now()로 set, 아니면 그대로*. atomic + race-free. SQLAlchemy 표준 패턴.
- DB-side `func.now()` 단일 시계 — Story 1.5에서 박힌 패턴(P10 patch 정합) 그대로 유지.

**Tutorial seen flag — AsyncStorage 단일 boolean key (per-device, per-logout-cycle)**

- 대안 1: backend `users.tutorial_seen_at` 컬럼 추가 — over-engineering(데이터 수집 의무 X + 마이그레이션 비용). yagni 회피.
- 대안 2: TanStack Query 캐시에 boolean — 메모리 휘발(앱 재시작 시 초기화). resume 흐름(AC2)에 적합 X.
- **채택**: AsyncStorage `bn:onboarding-tutorial-seen` (boolean stringified) — `signOut` 시 `clearAuth()`로 자동 정리(AC8). per-user-id 분리 X(같은 디바이스 1 사용자 가정 + signOut clear로 충분).
- **개인정보 격리**: 사용자 A 로그아웃 → 사용자 B 로그인 시, A의 seen 잔존이 B에게 노출되면 *B에게 tutorial 강제 스킵* → UX 저하(B는 정상 흐름으로 tutorial 봐야 함). `clearAuth()` 클리어로 B에게 fresh tutorial 노출 보장(AC8).

**Tutorial slide 수 — 3개 (epic AC line 497) — *2-3* 명시 범위 내**

- Epic AC: *"사용법 튜토리얼 2-3 화면"* + 3 핵심 메시지 명시(*"사진 한 장으로 식단 분석" / "한국 1차 출처 인용 피드백" / "디스클레이머 — 의학적 진단 아님"*).

### Anti-pattern 카탈로그 (본 스토리 영역)

[Source: Story 1.4·1.5 Anti-pattern 카탈로그, architecture.md#Anti-pattern (line 557-565)]

- ❌ Backend 신규 endpoint `POST /v1/users/me/onboarding/complete` 도입 — *튜토리얼 완료 시점 기록*은 PIPA·FDA audit 의무 X + `onboarded_at`은 profile POST 시 자동 set으로 충분. yagni 회피.
- ❌ `users.tutorial_completed_at` 컬럼 추가 — `onboarded_at`(Story 1.2 baseline)에서 *온보딩 흐름 전체 통과* 의미가 충분. tutorial은 정보 제공 화면 — 통과 여부 audit 의무 X.
- ❌ Tutorial seen flag를 backend 컬럼으로 — over-engineering. AsyncStorage(client-side) 충분. yagni.
- ❌ `react-native-swiper` 또는 `react-native-reanimated-carousel` 도입 — RN 표준 `ScrollView horizontal pagingEnabled`로 충분. 신규 의존성 + bundle 크기 증가 회피.
- ❌ Tutorial 슬라이드 카피를 backend `/v1/legal/tutorial`로 fetch — 정적 카피 + i18n NFR-L1 한국어 1차 → mobile 인라인 const 충분. backend trip 비용 + cold-start 지연 회피.
- ❌ `(tabs)/_layout.tsx` 4번째 가드 redirect 대상을 *조건부* 분기(`tutorialSeen ? /profile : /tutorial`) — (tabs) 가드는 server state(consent + profile_completed)에 의존, AsyncStorage fetch 대기 시 flicker. **채택**: 4번째 가드는 항상 `/tutorial`로 redirect, `tutorial.tsx` 자체가 seen flag 분기.
- ❌ `setTutorialSeen(true)`를 슬라이드 3 진입 시 자동 set — 사용자가 슬라이드 3 도중 앱 종료 시 *시작 안 한 채로 seen=true*로 마킹돼 다음 진입 시 tutorial 우회. **채택**: *시작* / *건너뛰기* 명시 탭 시점에만 set.
- ❌ `users.onboarded_at`을 `func.now()`로 무조건 update(`COALESCE` 안 쓰고) — 재 POST 시 *온보딩 첫 통과 시점* 의미 손실. Story 5.1 prefill 흐름에서 사용자 혼동(*첫 가입이 언제였는지* 표시 망가짐). **채택**: `func.coalesce(User.onboarded_at, func.now())`.
- ❌ tutorial 화면이 backend tutorial document version 추적 — 디스클레이머·약관·AD는 PIPA 의무로 version 추적, tutorial은 정보 제공 → version 추적 X.
- ❌ Web `/onboarding/tutorial/page.tsx` 도입 — architecture line 985 mobile-only 명시 정합 위반. yagni.
- ❌ AsyncStorage seen flag를 user_id 별 분리(`bn:onboarding-tutorial-seen-{user_id}`) — 단일 디바이스 1 사용자 가정 + signOut `clearOnboardingState()` clear로 충분. 복잡도 증가 회피.
- ❌ `secure-store.ts`에 AsyncStorage import해 `clearAuth()`에 tutorial-seen `removeItem` 추가 — `secure-store.ts:4` 주석(*"AsyncStorage / MMKV 평문 저장 금지 — NFR-S1 위반"*)이 코드베이스 강한 가이드라인. secure-store는 *토큰 전용* 책임 유지, tutorialState가 자체 `clearOnboardingState()` export + `signOut`에서 별도 호출 패턴 채택(책임 분리).
- ❌ AsyncStorage 신규 의존성 회피하고 *expo-file-system*(이미 standard) 또는 *useState 메모리*에 tutorial-seen 저장 — file-system은 무겁고 비표준 use case + useState는 앱 재시작 시 휘발(AC2 resume 흐름 부적합). AsyncStorage가 RN 생태계 표준 client preference store(Expo 공식 가이드 명시).
