## Story

As a **엔드유저**,
I want **나이·체중·신장·활동 수준·`health_goal` 4종 중 1개·식약처 22종 알레르기 다중 선택을 입력하기를**,
So that **시스템이 BMR/TDEE 계산과 알레르기 가드의 기반이 되는 내 건강 프로필을 갖게 된다.**

부수적으로 본 스토리는 (a) `users` 테이블에 건강 프로필 7개 컬럼을 0005 마이그레이션으로 추가하고, (b) `app/domain/allergens.py`에 식약처 별표 22종 enum + DB CHECK 제약을 박아 도메인 무결성(C4·NFR-C5)을 1지점에서 보장하며, (c) Story 1.3에서 deferred 된 `require_basic_consents` 라우터 wire(W14)를 `POST /v1/users/me/profile`에서 첫 적용한다. (d) `health_goal` Postgres ENUM(`users_health_goal_enum` 4값) + `activity_level` Postgres ENUM(`users_activity_level_enum` 5값)을 박아 KDRIs AMDR 매크로 룰 lookup(Story 3.5 fit_score) 및 BMR/TDEE 산출(Story 3.x)의 enum 기반을 마련한다. Story 1.6이 `(auth)/onboarding/automated-decision → tutorial → profile → /(tabs)` chain의 마지막 화면으로 본 스토리의 profile 화면을 끼우고 `users.onboarded_at`을 set한다 — 본 스토리는 `profile_completed_at`만 set하고 `onboarded_at`은 건드리지 않는다(scope 분리). Story 5.1이 본 스토리의 `POST /profile`을 `PATCH /v1/users/me`로 확장(수정 흐름) + Web 측 settings UI 도입.

1. **AC1 — 건강 프로필 입력 화면 6 필드 노출 + 폼 검증**

     2. **체중(kg)** — 0.1 단위 소수 입력(`keyboardType="decimal-pad"`), 1.0 ≤ weight_kg ≤ 500.0 검증
     3. **신장(cm)** — 정수 입력(`keyboardType="number-pad"`), 50 ≤ height_cm ≤ 300 검증

     5. **`health_goal`** — `weight_loss` / `muscle_gain` / `maintenance` / `diabetes_management` 4종 single-select(한국어 라벨: *체중 감량* / *근육 증가* / *유지* / *당뇨 관리*), 색상 + 텍스트 라벨 동시 표시
     6. **알레르기** — 식약처 22종 multi-select(체크박스 또는 chip 리스트). `domain/allergens.py:KOREAN_22_ALLERGENS` 와 1:1 매핑. 미선택 허용(*해당 없음* 의미 — `[]` 빈 배열 저장).

2. **AC2 — `POST /v1/users/me/profile` 백엔드 endpoint + UPSERT + `require_basic_consents` 게이트 wire**
   **Given** 동의 통과 사용자(JWT + 4종 동의 통과), **When** `POST /v1/users/me/profile` 호출(body: `{age: int, weight_kg: float, height_cm: int, activity_level: str, health_goal: str, allergies: list[str]}` snake_case 양방향), **Then**:

       - `age = body.age`, `weight_kg = body.weight_kg`, `height_cm = body.height_cm`, `activity_level = body.activity_level`, `health_goal = body.health_goal`, `allergies = body.allergies`
       - `profile_completed_at = now()` set (이미 set 된 row의 경우도 갱신 — *최신 프로필 기록 시점*. 과거 입력 history는 audit_logs Story 7.3 책임).
       - `updated_at = now()` 갱신(`Mapped onupdate=text("now()")` 자동 발화 — sync UPDATE statement는 ORM 이벤트 정상 작동, ON CONFLICT 분기 X 정합).
     - `Depends(require_basic_consents)` 게이트가 라우터에 wire — 4종 동의 미통과 사용자는 `BasicConsentMissingError` raise → RFC 7807 403 + `code=consent.basic.missing` (Story 1.3 W14 deferred 흡수).
     - 응답 status **200** + body `HealthProfileResponse`(AC3 모델). 본 endpoint는 *기존 row의 일부 필드 update* 의미가 강함(`users` row는 Story 1.2 OAuth 시점 이미 존재) — 신규 row 생성 케이스 X. 따라서 항상 200(Story 1.4 AC3 결정 정합).
     - 클라이언트가 보낸 값이 검증 실패 시 — Pydantic `ValidationError` → **400** RFC 7807 + `code=validation.error` (`main.py:validation_exception_handler` 매핑, Story 1.2/1.3/1.4 정합):
       - `age` 정수 1~150 범위 외 → 400
       - `weight_kg` 1.0~500.0 범위 외 또는 NaN/inf → 400
       - `height_cm` 정수 50~300 범위 외 → 400
       - `activity_level` 5 enum 값 외 → 400
       - `health_goal` 4 enum 값 외 → 400

       - `allergies` 배열 길이 22 초과 → 400 (`Field(max_length=22)` — 도메인 상한 + DoS 차단).
       - `weight_kg`이 NaN/Infinity → 400 (`Field(allow_inf_nan=False)` — DB Numeric DataError 회피).
       - `weight_kg`이 소수 둘째 자리 이상 (예: 70.55) → 400 (`field_validator` — DB `Numeric(5,1)` silent rounding 차단).

3. **AC3 — `HealthProfileResponse` 응답 모델 + `GET /v1/users/me/profile` endpoint**
   **Given** 인증 + 동의 통과 사용자, **When** `GET /v1/users/me/profile` 호출, **Then**:
     - 응답 status **200** + body `HealthProfileResponse`:
       ```python
       class HealthProfileResponse(BaseModel):
           age: int | None
           weight_kg: float | None  # Decimal → float 직렬화 (snake_case 양방향)
           height_cm: int | None
           activity_level: ActivityLevel | None  # Literal["sedentary", "light", "moderate", "active", "very_active"]
           health_goal: HealthGoal | None        # Literal["weight_loss", "muscle_gain", "maintenance", "diabetes_management"]
           allergies: list[str]                  # 22종 부분집합. NULL → [] 빈 배열로 응답(클라이언트 분기 단순화)
           profile_completed_at: datetime | None
       ```
     - `users.profile_completed_at IS NULL`(미입력) → 6 필드 모두 NULL + `allergies=[]` + `profile_completed_at=null`. **404 미사용** — 사용자는 항상 존재(인증 통과한 사용자는 `users` row 존재 보장), *프로필 미입력*은 *row 없음*과 의미 다름.
     - `Depends(current_user)`만 wire(인증만 충분, 동의 게이트 X) — 사용자는 자기 정보 조회는 동의 무관(Story 1.3 deps.py 주석 정합 *"`/v1/users/me` 같은 자기 정보 조회 endpoint에 적용 금지"*). 본 GET은 같은 정책. **PIPA Art.35 근거**: *정보주체의 권리(열람·정정·삭제·처리정지)*는 동의 철회와 독립된 *권리*로 보장 — 자기 데이터 *조회*를 동의 게이트로 차단하면 권리 행사 자체가 봉쇄되고 UX 모순 발생(자기가 입력한 데이터를 볼 수 없음). 따라서 *작성* 경로(POST/PATCH/DELETE)는 동의 게이트 wire, *조회* 경로(GET)는 인증만 wire — 본 결정은 향후 외부 보안 audit 시 재검토.
     - **본 endpoint 도입 이유**: 모바일/Web이 진입 시 `profile_completed_at` boolean 분기에 의존 — `GET /v1/users/me` 응답 모델에 `profile_completed_at` 추가만으로도 가능하나, 프로필 *값* prefill(Story 5.1 수정 흐름)을 본 스토리에서 forward-hook 박아두면 1번의 endpoint로 prefill + 분기 양쪽 사용 가능.

4. **AC4 — `users` 테이블 7 컬럼 추가 + Postgres ENUM 2종 + DB CHECK (0005 마이그레이션)**
   **Given** 본 스토리 시작 시점 `users` 테이블(0002 baseline + Story 1.2 컬럼), **When** `alembic upgrade head` 실행, **Then** `users` 테이블에 컬럼 7개 ADD + Postgres ENUM 타입 2개 CREATE + CHECK 제약 1건 ADD:
     | 컬럼 | 타입 | 제약 |
     |------|-----|-----|
     | `age` | `integer` | NULL, `CHECK (age IS NULL OR (age >= 1 AND age <= 150))` |
     | `weight_kg` | `numeric(5,1)` | NULL, `CHECK (weight_kg IS NULL OR (weight_kg >= 1.0 AND weight_kg <= 500.0))` |
     | `height_cm` | `integer` | NULL, `CHECK (height_cm IS NULL OR (height_cm >= 50 AND height_cm <= 300))` |
     | `activity_level` | `users_activity_level_enum` | NULL |
     | `health_goal` | `users_health_goal_enum` | NULL |
     | `allergies` | `text[]` | NULL, `CHECK (allergies IS NULL OR allergies <@ ARRAY[...22종...]::text[])` |
     | `profile_completed_at` | `timestamptz` | NULL |
   - Postgres ENUM 타입 2개 CREATE:
     - `CREATE TYPE users_activity_level_enum AS ENUM ('sedentary', 'light', 'moderate', 'active', 'very_active')` (architecture line 396 naming `<table>_<col>_enum` 정합)
     - `CREATE TYPE users_health_goal_enum AS ENUM ('weight_loss', 'muscle_gain', 'maintenance', 'diabetes_management')` (architecture line 291 명시)
   - **22종 도메인 CHECK 제약**: SQL 리터럴 배열에 22 식약처 별표 표시기준 22종을 ARRAY로 인라인. PostgreSQL `<@` (subset) 연산자로 *입력 배열의 모든 원소가 22종 정의 집합의 부분집합인지* 검증. 빈 배열 `[]`은 기본 통과(빈 집합은 모든 집합의 부분집합) — *알레르기 없음* 의미. 단, 22종 문자열 정의는 `app/domain/allergens.py:KOREAN_22_ALLERGENS`와 1:1 동일 — 마이그레이션에서 직접 재사용(파이썬 모듈 import해서 SQL 리터럴 생성, 또는 알렘빅 revision 작성 시점에 module 식별자로 fetch 후 텍스트 인라인). **결정**: 마이그레이션 파일 상단에 `from app.domain.allergens import KOREAN_22_ALLERGENS` import + SQL 인라인 생성. 향후 22종 변경 시 `app/domain/allergens.py`만 수정 + 새 alembic revision으로 CHECK 제약 drop+recreate(R8 SOP, Story 8 운영 cron `scripts/verify_allergy_22.py` 정합).
   - **인덱스 추가 OUT** — 본 스토리 쿼리 패턴은 user_id 기준 단일 row lookup만(`users.id` PK 자동 활용).
   - **Story 1.2/1.3/1.4 회귀**: 신규 컬럼·ENUM·CHECK 모두 nullable·NULL 허용 → 기존 INSERT/UPDATE statement(`auth.py` user 생성, `consents.py` row 생성) 회귀 0건. 기존 테스트 `truncate users RESTART IDENTITY CASCADE`도 컬럼 추가 영향 X(전체 row drop).
   - SQLAlchemy 모델 `app/db/models/user.py`에 컬럼 7개 + ENUM 타입 매핑 추가 — `Mapped[int | None]` / `Mapped[Decimal | None]` / `Mapped[ActivityLevel | None]`(import enum) 패턴(Story 1.4 컬럼 패턴 정합). 모듈 docstring 1줄 갱신(*"Story 1.5가 건강 프로필 7컬럼 추가"*).
   - `alembic upgrade head` 로컬 검증 → 컬럼 7개 + ENUM 2개 + CHECK 제약 추가 확인. `alembic downgrade -1` → upgrade 재실행 idempotency 검증(downgrade는 ENUM TYPE DROP 시 dependent column이 없는 상태여야 — 따라서 컬럼 drop 먼저 → ENUM TYPE drop 순서 엄격 보존).

5. **AC5 — `app/domain/allergens.py` SOT 모듈 + 22종 frozenset + helper**
   **Given** 본 스토리, **When** `app/domain/allergens.py` 신규 생성, **Then**:
     ```python
     from typing import Final

     # 식약처 *식품 등의 표시기준 별표* 22종 알레르기 유발물질 (research 2.4 기준).
     # SOT — 변경 시: (a) 본 tuple 갱신 → (b) 새 alembic revision으로 users.allergies CHECK 제약 drop+recreate
     # → (c) `scripts/verify_allergy_22.py`(R8 SOP, Story 8) 분기 1회 cron 실행 결과 동기.
     KOREAN_22_ALLERGENS: Final[tuple[str, ...]] = (
         "우유", "메밀", "땅콩", "대두", "밀",
         "고등어", "게", "새우", "돼지고기", "아황산류",
         "복숭아", "토마토", "호두", "닭고기", "난류(가금류)",
         "쇠고기", "오징어", "조개류(굴/전복/홍합 포함)", "잣", "아몬드",
         "잔류 우유 단백", "기타",
     )
     KOREAN_22_ALLERGENS_SET: Final[frozenset[str]] = frozenset(KOREAN_22_ALLERGENS)


     def is_valid_allergen(value: str) -> bool:
         """단일 알레르기 라벨이 22종 도메인 내 인지 확인 — Pydantic validator/route 검증에서 사용."""
         return value in KOREAN_22_ALLERGENS_SET


     def normalize_allergens(values: list[str]) -> list[str]:
         """22종 부분집합 검증 + dedup. invalid 항목 발견 시 ValueError 발생.

         반환 순서는 KOREAN_22_ALLERGENS 정의 순(결정성). dedup은 frozenset 거친 후 정의 순 재정렬.
         """
         seen = set()
         for v in values:
             if v not in KOREAN_22_ALLERGENS_SET:
                 raise ValueError(f"unknown allergen: {v!r}; must be one of 22 식약처 standard.")
             seen.add(v)
         return [a for a in KOREAN_22_ALLERGENS if a in seen]
     ```
   - **22종 정확 인용**: research 2.4(line 170-174) 정합. 정의 순서는 식약처 표시기준 별표 PDF 순(우유→메밀→땅콩→… 순). dev agent는 이 순서를 *바꾸지 말 것* — 마이그레이션 CHECK 제약과 enum-like ordering이 동일해야 회귀 차단.
   - **카테고리 vs 라벨 분리는 본 스토리 OUT** — 영문 카테고리(milk/peanut 등) 분리는 i18n 정책 Story 8.x 시점 결정. 본 스토리는 한국어 라벨만 SOT로 유지.
   - **`scripts/verify_allergy_22.py` 도입은 Story 3.1/Story 8 책임** — 본 스토리는 SOT 모듈 + 단위 테스트만(아래 AC10).

    - **결정 — SQL 인라인 vs alembic 전용**: 모델 선언에서 CHECK 제약 SQL을 인라인하면 SQLAlchemy autogenerate가 검출 가능. 다만 22종 변경 시 마이그레이션과 동기 강제. **선택**: 모델 선언에 `KOREAN_22_ALLERGENS` 인라인 + alembic 0005 마이그레이션이 동일 SQL ADD. autogenerate diff 0건 보장. 22종 갱신 시 둘 다 수정 + 새 alembic revision으로 drop+recreate(R8 SOP).

**Postgres ENUM `users_activity_level_enum` / `users_health_goal_enum` 도입 (Postgres ENUM vs CHECK + text)**

- architecture.md line 291 명시: `health_goal` Postgres ENUM. 본 스토리는 같은 패턴으로 `activity_level`도 ENUM 도입.
- ENUM의 장점: (1) DB-level 도메인 무결성(잘못된 값 INSERT 거부, 22종 알레르기 CHECK와 동등 보호), (2) Postgres ENUM은 sort 순서가 정의 순(KDRIs lookup 시 결정성), (3) 4-byte OID 저장 → text 비교 대비 약간 빠름.
- ENUM의 단점: (1) 값 추가/제거 시 마이그레이션 필요(`ALTER TYPE ... ADD VALUE`), (2) Postgres-only(MySQL 등 호환성 X). 본 프로젝트는 Postgres 17 박힘(architecture line 285) → 단점 무관.
- **결정**: text + CHECK 대비 Postgres ENUM 채택. 22종 알레르기는 *값 갱신 빈도가 분기 1회*라 CHECK 제약(text[])이 더 가벼움 — array 타입에 ENUM 적용은 Postgres 표준이지만 마이그레이션 복잡도 + 22종 갱신 시 ENUM `ADD VALUE` 후 `<@` 비교가 자동 갱신되지 않음 → CHECK + text[] 채택. health_goal·activity_level은 4·5종 *고정* — ENUM 적합.

**`weight_kg` Decimal vs float — float wire format + DB Numeric(5,1) 채택**

[Source: architecture.md#Format Patterns (line 478-481)]

- JSON wire format은 float(JSON `number` 표준). DB 컬럼은 `numeric(5,1)` (소수 한 자리 정밀도 보장 + 1.0~999.9 범위).
- IEEE 754 double 53-bit mantissa로 0.1 단위 ±500 표현 정확도 충분 — Decimal 도입 ROI 0.
- 자동 변환: Pydantic float → SQLAlchemy Numeric 자동 변환 작동. SELECT 시 SQLAlchemy `Decimal` 반환 — `_build_profile_response`에서 `float(user.weight_kg)` 명시 변환으로 wire format 정합.
- 클라이언트 0.1 step 입력 — mobile `keyboardType="decimal-pad"` + zod `step(0.1)` 또는 `multipleOf(0.1)` 검증. Web `<input type="number" step="0.1">` 검증.

### Anti-pattern 카탈로그 (본 스토리 영역)

[Source: Story 1.3·1.4 Anti-pattern 카탈로그, architecture.md#Anti-pattern (line 557-565)]

- ❌ 22종 알레르기를 *Postgres ENUM*으로 도입 — 분기 1회 갱신 빈도 + ENUM `ADD VALUE` 후 `<@` 비교 자동 갱신 X → text[] + CHECK 제약 채택.
- ❌ `health_goal` / `activity_level`을 *text + CHECK 제약*으로 도입 — 4·5종 고정값 + 자주 호출되는 lookup → Postgres ENUM 채택(architecture line 291 명시).
- ❌ `weight_kg`을 `float(8)` 또는 `double precision`으로 도입 — DB 정밀도 보장 X. `numeric(5,1)` 채택(0.1 단위 정확).
- ❌ `users.profile_completed_at` 대신 `users.onboarded_at` 통합 사용 — Story 1.6 의미와 분리 필수(*프로필 입력* vs *온보딩 흐름 통과*).
- ❌ `POST /me/profile` 응답에 `profile_completed_at: bool` 단순 boolean — `datetime | None` 채택(Story 5.1 prefill 시 *언제 마지막 입력*을 클라이언트가 활용 가능 + audit 흔적).
- ❌ `require_basic_consents`를 `GET /me/profile`에 wire — 자기 정보 *조회*는 동의 무관(Story 1.3 deps.py 주석 정합). wire 시 사용자가 미동의 상태로 자기 프로필 조회 불가 → UX 문제. *POST에만* wire.
- ❌ 22종 알레르기 라벨을 *영문화* (예: `"milk"` 대신 `"우유"`) — i18n 정책 NFR-L1~L4 정합 한국어 1차. 영문 라벨은 Story 8.x i18n 도입 시점.
- ❌ `allergies = []` 응답에 NULL 노출 — 클라이언트 분기 단순화 위해 빈 배열로 응답(`_build_profile_response`에서 `user.allergies or []`).

- ❌ Pydantic `weight_kg` 필드를 `Decimal` 타입으로 받기 — JSON wire format float 표준 정합 X. `float` 채택.
- ❌ `(tabs)/_layout.tsx`의 4번째 가드를 (1) auth 다음 (2번째)로 배치 — basic/AD 미통과 사용자가 profile 화면으로 우회됨. 정확 순서 (1)→(2)→(3)→(4) 유지.

### Debug Log References

- **타입 mismatch — `varchar[] vs text[]`**: 초기 모델에서 `allergies` 컬럼을
  `ARRAY(String)`로 선언 → SQLAlchemy가 `varchar[]` 생성 → CHECK 제약 SQL의
  `ARRAY[...]::text[]`와 비교 시 PostgreSQL `operator does not exist:
  character varying[] <@ text[]` 오류. 해결: `ARRAY(Text)`로 변경 → `text[]`로 일치.
  alembic 트랜잭션 안에서 실패해 자동 rollback, 다음 시도에서 정상 적용.
- **Web ESLint `react-hooks/refs` 위반**: `inflightRef` + `abortRef`를
  `handleSubmit(async ...)` 콜백 내에서 access 시 false-positive로 차단.
  react-hook-form은 자체 `formState.isSubmitting`으로 double-submit 방지하므로
