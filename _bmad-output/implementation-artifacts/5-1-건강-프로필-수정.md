## Story

As a 엔드유저,
I want 내 건강 프로필(나이/체중/신장/활동/`health_goal`/22종 알레르기)을 모바일 또는 Web 설정 화면에서 언제든 수정하고 변경이 이후 모든 분석에 즉시 반영되기를,
so that 체중 변화·목표 전환·알레르기 추가 등 라이프 변화를 분석에 곧바로 통합할 수 있다.

### 결정 노트

#### D1 — `PATCH /v1/users/me` vs `PATCH /v1/users/me/profile`

- **선택 (`PATCH /me/profile`)**: 기존 `GET/POST /v1/users/me/profile` 정합. POST/GET/PATCH 같은 path scope의 verb-only 구분이 *직관 + 검색 가능*. Story 4.4 `PATCH /me/macro_goal`도 sub-resource 분리 패턴이라 통일성. `/v1/users/me` PATCH는 미래 *display_name/picture_url/email* 같은 *계정 metadata* 수정 슬롯으로 유지(현재 미사용 — Epic 7 admin 흐름이 들어올 가능성).
- **Reject (`PATCH /me`)**: epics.md:788 표기 그대로지만 *생략형* 해석. `/me`가 *계정 entity 전체*를 PATCH하면 health profile + macro_goal + notification_settings 다중 sub-resource 혼재 → body shape 비대해지고 backend dispatch 복잡. REST sub-resource 분리가 OpenAPI codegen 클라이언트에도 깔끔.

#### D2 — Mobile/Web 폼 컴포넌트 재사용 vs 중복

- **Mobile (선택: 추출)**: Story 1.5 onboarding profile.tsx의 폼 markup(6 필드 Controller + 22 알레르기 chip + styles 50+ 줄) 중복은 22종 변경 시점에 양쪽 sync 강제 → drift 위험 최고. `HealthProfileEditForm` props로 mode/initial 분기는 단일 SOT 가독성 우위. **단, Story 1.5 회귀 0건 강제** — Task 4.2에서 onboarding profile.tsx의 폼 동작/markProfileCompleted callback/redirect chain invariant 검증.
- **Web (선택: 중복 허용)**: web 폼 markup은 Server Component 경계 + props 인터페이스 차이(`onboarding`은 redirect chain의 일부, settings는 SSR prefill props)로 추출 가치 < 중복 가독성 우위. 단, **schema/labels/constants는 단일 SOT**(`features/onboarding/healthProfileSchema.ts`) — markup만 중복. 22종 변경 시 schema 한 지점만 갱신.

#### D3 — `profile_updated_at` 신규 컬럼 vs `updated_at` 재사용

- **선택 (신규 컬럼)**: 사용자 프로필을 *명시적으로 수정한 시점*은 macro_goal/notification 변경과 *분리* 의미. `updated_at`은 *행 단위 변경 시점*(macro_goal patch도 bump). 사용자 *프로필* timestamp는 별 컬럼이 의미 명확 + 미래 *마지막 프로필 수정 N일 전* 같은 분석 쿼리 단순화. alembic 1건 추가는 비용 < 가치.
- **Reject (`updated_at` 재사용)**: 의미 혼재 → 분석 쿼리에서 *어떤 변경*이 trigger인지 모호. *마지막 프로필 수정 1년 전* 같은 inactive 사용자 nudge에 잘못된 timestamp 사용 위험. 분리가 SOT 정합.

#### D4 — `null` 명시 송신 = NULL set vs key 무시

- **선택 (NULL set)**: 6 필드 모두 *원래 nullable*(Story 1.5 baseline) — `null` 송신은 *명시 clear* 의미. PATCH `{"weight_kg": null}` → `users.weight_kg = NULL`(체중 미설정 상태로 reset). 사용자가 *내 프로필을 미입력 상태로 되돌리고 싶다* 같은 액션은 드물지만 권리 정합.
- **Reject (key 무시)**: `model_dump(exclude_unset=True)`로 *명시 송신*만 처리하면 `null`도 *명시 송신*으로 분류 → NULL set 분기 진입. `exclude_none=True` 사용 시 `null` 키가 dict에서 제거되어 *부재*와 동등 처리되지만 REST PATCH 표준은 `null`을 *명시 clear*로 해석. 본 스토리는 표준 정합 + Story 4.4 macro_goal `null` 송신 = key 삭제 패턴 정합.

#### D5 — Web settings index 페이지 신규 vs `/settings/profile` 단독

- **선택 (단독)**: Web에는 *별 settings index*가 미존재(현재 `/settings/macro-goal`만 단독). 본 스토리도 `/settings/profile` 단독 페이지. *별 settings index 페이지* 신규는 OUT — Story 8.6 polish 시점(영업 자료 패키지 + Web UX 통합)에 일괄 도입. 사유: ① 1인 8주 정합 ② 영업 데모 시나리오는 모바일 위주 ③ Web settings 영역은 admin/dashboard 강조라 사용자 settings는 진입 빈도 낮음 — 직접 URL/dashboard 마이크로카피 fallback 충분.
- **Reject (index 페이지 동시 신규)**: scope 폭증 + macro-goal 메뉴 구조 디자인 결정 필요(Tabs/Sidebar/List 등). Story 8.6에 통합이 SOT 정합.

#### D6 — `HealthProfilePatchRequest` `at-least-one` 가드 — Pydantic vs SQL

- **선택 (Pydantic)**: Story 4.4 `MacroGoalPatchRequest` `model_fields_set` 기반 SOT 정합 — *명시 송신* 필드만 가드. partial PATCH 회귀 학습(빈 body 거부 + 다중 PATCH 누적 송신은 set만 처리) 정합.
- **Reject (SQL)**: SQL UPDATE에서 `WHERE` 절로 *모든 필드 NULL인 update 차단*은 비현실적. PATCH 시 *값 변경*과 *부재*를 구분하는 SQL semantics는 복잡. Pydantic 1차 게이트가 SOT.

#### D7 — Audit log 의무 vs 자기 수정 제외

- **선택 (제외)**: epics.md:792 명시 + PIPA Art.35 권리 정합. system log(structlog `users.profile.patched`)는 운영 모니터링 SOT — audit_log 테이블과 분리. 자기 수정은 *자기 데이터 권리* — admin 수정만 audit 의무(FR37 정합).
- **Reject (포함)**: audit_log 테이블에 자기 수정도 기록하면 *PIPA grace period 30일 후 cascade 파기*에서 의도치 않은 metadata 영속화. 사용자 권리(처리정지) 행사 시 audit 의무와 충돌 가능. 분리가 SOT.

async def patch_health_profile(
    body: HealthProfilePatchRequest,
    db: DbSession,

    fields_dict = body.model_dump(exclude_unset=True)

        update(User)
        .where(User.id == user.id)
        .values(
            **fields_dict,
            profile_updated_at=func.now(),
            updated_at=func.now(),
            # profile_completed_at 미터치 — Story 1.5 invariant 보존.

        .returning(User)

    )
    updated_user = result.scalar_one()
