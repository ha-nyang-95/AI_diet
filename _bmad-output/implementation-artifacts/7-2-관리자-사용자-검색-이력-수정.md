## Story

As a 관리자,
I want Web 관리자 화면에서 사용자를 검색하고 식단 이력·피드백 로그·프로필을 조회·수정하기를,
So that 사용자 문의 응대 및 데이터 거버넌스 운영이 가능하다.

     ```python
     """민감정보 마스킹 helper — NFR-S5 SOT (Story 7.2).

     architecture.md:309 *"Sentry before_send hook + structlog processor — 정규식으로

     Story 7.4에서 `unmask=True` flag 추가 시 plaintext 반환 분기 — 그때 본 모듈에 흡수.
     """
     from decimal import Decimal


     def mask_email(email: str) -> str:
         """`j***@example.com` 패턴. Story 7.1 `auth.py:_mask_email` SOT 동형 — 본 스토리에서
         `auth.py`의 helper를 본 모듈로 *이동*(7.1 owner의 SOT 위치 변경 + auth.py에서 import).
         """
         if "@" not in email:
             return "***"
         local, _, domain = email.partition("@")
         if len(local) <= 1:
             return f"***@{domain}"
         return f"{local[0]}***@{domain}"


     def mask_weight(weight_kg: Decimal | float | None) -> str | None:
         """체중 마스킹 — `**kg` (값 자체 노출 0). None 입력 시 None pass-through.
         원문 보기 토글(7.4)은 `f"{value:.1f}kg"` 분기.
         """
         return None if weight_kg is None else "**kg"


     def mask_height(height_cm: int | None) -> str | None:
         """신장 마스킹 — `**cm`. 동일 패턴."""
         return None if height_cm is None else "**cm"


     def mask_allergies_count(allergies: list[str] | None) -> str | None:
         """알레르기 *상세 항목명*은 마스킹, *건수*는 운영 컨텍스트로 노출.

         - None: 미입력 — null 응답.
         - []: 등록 없음 — `*** 등록 없음`.
         - [...]: `*** N건 등록`.

         원문 보기 토글(7.4)은 actual list 반환 분기.
         """
         if allergies is None:
             return None
         if not allergies:
             return "*** 등록 없음"
         return f"*** {len(allergies)}건 등록"
     ```

**대안 B — `app/core/masking.py` 신규 + 4 helper 통합 + `_mask_email` → `mask_email`(public) 이동** *(채택)*: NFR-S5 SOT 단일 지점 + Sentry hook + structlog processor + admin endpoint 모두 동일 import. Story 7.4 *원문 보기* 토글 시 `unmask=True` flag 추가도 본 모듈에서 일괄(다중 수정 회피).

#### 마스킹 정책 — 기본 마스킹만 (원문 보기 토글은 7.4 forward)

본 스토리는 *FR39 기본 마스킹*만 구현 — *원문 보기 토글*(`?include_pii=true` 또는 별 endpoint `POST /admin/users/{user_id}/pii-reveal`)은 Story 7.4 책임. 사유:
- *원문 보기 액션 자체*가 audit log에 `action="user_pii_view"` 기록 의무(FR39) — Story 7.3 audit 인프라 선행 의무.
- *5분 비활동 자동 복원* 흐름은 Web client-side 타이머 + cookie/localStorage SOT 분리 검토 의무 — Story 7.4 UI 책임 통합.

#### 검색 SQL — UNION 회피, OR 분기 single query

**대안 A — `UNION ALL` of email-match SELECT and id-prefix SELECT**: 거부함. 두 분기에서 같은 row hit 시 dedup 의무 + cursor pagination keyset이 두 SELECT에 걸쳐 broken (each SELECT가 독립 정렬).

**대안 B — `OR` 단일 쿼리 + Postgres planner choose** *(채택)*: planner가 `idx_users_email_lower` hit 분기 + UUID prefix full scan 분기 *bitmap OR* 조합 — 1만 row sub-second(EXPLAIN 검증은 Story 8 perf hardening). dedup 자동(같은 row 1회 hit). cursor pagination keyset 단일 정렬(`created_at DESC, id DESC`)으로 일관.

#### UUID prefix 검색 — 인덱스 미생성

**사유**: `users.id`는 `gen_random_uuid()` 결과 — UUID v4 random distribution이라 prefix(`first 8 chars`) 검색은 어떤 prefix도 균등 분포. btree index가 prefix range scan을 hit해도 *행 데이터가 무작위 분포된 페이지*에 흩어져 random I/O가 full scan보다 느림. 1만 row × text cast ≤ 100ms 실측 가정으로 full scan 허용. 100만 row 이상 + UUID prefix 검색 빈도 폭증 시 trigram(`pg_trgm` extension + GIN index) 도입 검토 — Story 8 perf hardening.

#### admin 식단 삭제 — soft delete만 (Story 2.1 정합)

**대안 A — admin은 hard delete 가능**: 거부함. soft delete invariant(30일 grace + `soft_delete_purge` cron worker — Story 5.2)를 admin이 우회하면 data lifecycle 정합 위반 + 사용자 권리(PIPA Art.35 정정/삭제) 모순(사용자 자기 식단 삭제도 soft인데 admin이 hard면 거버넌스 비대칭).

**대안 B — soft delete만** *(채택)*: Story 2.1 `delete_meal` 패턴 1:1 — admin은 *대상 user_id*만 다르고 흐름 동일. 30일 grace 후 `soft_delete_purge` cron이 자동 cascade.

#### `require_basic_consents` admin endpoint *미적용*

**사유**: 4종 동의(disclaimer/terms/privacy/sensitive_personal_info)는 *자기 데이터* 처리 진입 게이트(Story 1.3 W14 SOT). admin이 *타 사용자* 데이터를 거버넌스 컨텍스트로 처리하는 것은 PIPA 18조 *처리 위탁* 또는 *서비스 운영 정합*이라 별 동의 흐름. admin 자기 자신의 4종 동의는 OAuth 시점에 받은 상태이지만, 대상 사용자의 동의 상태는 admin 행위에 무관(사용자가 동의 철회한 후에도 데이터 본인이 삭제 요청하면 admin이 처리 가능 — Story 5.2 회원 탈퇴 흐름 정합).

#### `profile_updated_at` set 정책

admin이 타 사용자 프로필 수정 시에도 `profile_updated_at = now()` set — Story 5.1 SOT 정합(*프로필이 마지막으로 수정된 시점* — *누가 수정했는지* 무관). `profile_completed_at` 미터치 + `onboarded_at` 미터치 invariant도 동일. 사용자가 다음 로그인 시 *내 프로필이 누군가에 의해 수정됨*을 가시화하기 위해 별도 `profile_updated_by_admin_at` 컬럼 추가는 Story 7.3 audit log SOT로 일괄 흡수(컬럼 폭발 회피).

#### Web admin API fetch 패턴 — `adminApiFetch` 별 helper

**대안 A — `apiFetch` 그대로 admin 호출**: 거부함. `apiFetch`의 401 → `/v1/auth/refresh` 자동 재시도 흐름은 user JWT refresh — admin JWT는 refresh 미지원이라 무한 redirect 또는 user `/login` 화면으로 이탈(admin 흐름이 user login에 빠짐).

**대안 B — `adminApiFetch` 별 helper** *(채택)*: 401 → `/admin/login?error=expired` 즉시 redirect, 403 → `/admin/login?error=forbidden` (role 박탈 또는 IP 가드 차단). cookie path=`/` 자동 첨부(Story 7.1 CR 정합) — `credentials: "include"`만으로 충분.

#### 전부 SSR fetch vs Client Component fetch 분기

**상세 페이지(`/admin/users/[user_id]`)**: SSR initial fetch — `cookies()`로 admin token 추출 + `Authorization: Bearer` 헤더 forward. *cross-origin server-side fetch는 cookie 미첨부* (도메인 격리). Web client-side는 `adminApiFetch`로 cookie 자동 첨부.

**검색 페이지(`/admin/users`)**: Client Component fetch only — 검색은 사용자 입력 후 trigger되는 동적 액션이라 SSR 불필요 + TanStack Query stale-while-revalidate 패턴이 더 정합. 빈 상태로 SSR 후 hydration 시 검색 폼만 노출.
