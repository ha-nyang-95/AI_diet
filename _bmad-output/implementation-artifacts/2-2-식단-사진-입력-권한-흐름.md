## Story

As a **엔드유저**,
I want **식단을 사진으로 촬영하거나 갤러리에서 선택해 업로드하고, 권한 거부 시에도 빈 화면이 아닌 안내를 받기를**,
So that **손쉬운 시각 기록 + 권한 거부 후에도 서비스가 막히지 않는 완성도를 경험한다.**

본 스토리는 Epic 2의 두 번째 스토리로, **(a) 백엔드 R2 presigned URL adapter + `POST /v1/meals/images/presign` endpoint**, **(b) `meals.image_key` nullable 컬럼 추가 (0007 마이그레이션)**, **(c) 모바일 `expo-image-picker` + 카메라/사진첩 권한 hook + `PermissionDeniedView` 표준 컴포넌트**, **(d) `MealInputForm`의 사진 첨부 UI(카메라/갤러리 버튼 + 썸네일 미리보기) + R2 직접 업로드 helper**의 baseline을 박는다.

부수적으로 (a) `raw_text` NOT NULL invariant는 그대로 유지 — 사진-only 입력 시 서버가 `"(사진 입력)"` placeholder로 fallback (Story 2.3 OCR이 추출 결과로 덮어씀), (b) R2 read는 *public CDN URL* 직접(architecture line 868 정합 — 인증 토큰 불필요) — signed read URL 도입은 Story 8 hardening, (c) image content moderation / EXIF 제거 / thumbnail 생성은 본 스토리 OUT(Story 8 polish 또는 Growth), (d) OCR 추출 흐름(`parsed_items` 확인 카드)은 **Story 2.3 책임** — 본 스토리는 *업로드 + image_key 저장 + 응답 image_url* baseline까지, (e) 오프라인 큐잉은 *사진 OUT*(epic 2.5 명시 — *"사진은 온라인에서 처리 가능"*) — 사진 입력은 온라인 필수. 본 스토리 종료 시 Epic 2 두 번째 스토리(2.2) 완료 → Story 2.3 OCR이 `image_key`를 입력으로 활용 가능.

5. **AC5 — `PATCH /v1/meals/{meal_id}` image_key 수정 (require_basic_consents)**
   **Given** 4종 기본 동의 통과 사용자 + 자기 소유 식단, **When** `PATCH /v1/meals/{meal_id}` body `{"image_key": "meals/<user_id>/<new_uuid>.jpg"}` 호출, **Then**:
     - HTTP 200 + 응답 body `MealResponse` (갱신 후 9 필드).
     - DB 업데이트: `image_key` 갱신 + `updated_at` 자동 갱신 (Story 2.1 P5 정합 — `onupdate=text("now()")`).
     - **요청 body 스키마 갱신** (`MealUpdateRequest`, `extra="forbid"`):
       - `raw_text: str | None = None` (Story 2.1 정합 — D4: absent ≡ explicit null ≡ no-op).
       - `ate_at: datetime | None = None` (Story 2.1 정합).
       - `image_key: str | None = Field(default=None, max_length=512)` — 신규 추가. **명시 `null` vs 키 부재 시맨틱 — 키 부재(no-op) / 명시 `null`(image_key 클리어 — 사진 제거)**:
         - 본 스토리는 **명시 `null` = 클리어** 채택 — 사용자가 사진 없이 텍스트만 남기고 싶을 때 PATCH 호출. Pydantic 기본 absent vs explicit null 구분은 `model_fields_set`(Pydantic v2 표준). 라우터에서 `"image_key" in body.model_fields_set` 체크 → `update().values()` dict에 `image_key=body.image_key`(즉 None) 명시 포함. absent이면 dict에 image_key 키 자체 미포함 → SQL 미변경.
       - `@model_validator(mode="after") _at_least_one_field` — Story 2.1 정합 갱신. **Pydantic `model_fields_set`** 사용 (값이 아닌 *송신 여부* 검사):
         - `if not self.model_fields_set.intersection({"raw_text", "ate_at", "image_key"}): raise ValueError("at least one field required")`.
         - 사유: `raw_text=null` (D4 — no-op) 또는 `image_key=null` (클리어) 송신은 *각각의 의미가 있는* 필드 송신 — 단순 `is None` 검사로는 의미 손실. Story 2.1 validator는 `is None` 사용했으나 image_key explicit-null clear 시맨틱과 호환하려면 `model_fields_set`로 갱신 필수.
     - **소유권 검증 — image_key 갱신 시 추가 가드**: 라우터 inline — `if body.image_key is not None and not body.image_key.startswith(f"meals/{user.id}/"): raise MealImageOwnershipError(400)`. (AC4와 동일 패턴). `image_key=None`(클리어)은 검증 대상 X.
     - **단일 SQL 패턴 유지** — Story 2.1 race-free `update().where(id, user_id, deleted_at IS NULL).values(...).returning(Meal).execution_options(populate_existing=True)` 그대로. invariant 추가 검증 X — `raw_text` NOT NULL 컬럼 + Story 2.1 D4 시맨틱(`raw_text=null` 송신은 no-op)으로 *PATCH 후 raw_text=None + image_key=None 상태는 SQL 레벨에서 도달 불가* (NOT NULL 위반 BEFORE column-level — DB가 reject). 즉 *invariant 위반은 reachable 0건* — 별도 가드 도입 회피(YAGNI). 추가 SELECT round-trip 0건.
     - HTTP status·에러: Story 2.1 정합 (404 일원화 / 401 / 403 / 400).

## Dev Notes

### Architecture decisions (반드시 따를 것)

[Source: architecture.md#Data Architecture (line 281-299), architecture.md#API Patterns (line 314-326), architecture.md#Architectural Boundaries (line 845-868), architecture.md#Integration Points (line 899-927), epics.md#Story 2.2 (line 520-533), Story 2.1 Dev Notes 패턴]

**`meals.image_key` nullable 컬럼 — `raw_text` NOT NULL invariant 유지 + 서버 placeholder fallback**

- 0007 마이그레이션은 **`image_key TEXT NULL` 1 컬럼만** 추가. `raw_text`는 *NOT NULL 유지*(Story 2.1 design intent line 25 — *"사진 입력 도입 후에도 NOT NULL 유지 — 사진은 별도 image_key 컬럼 + 텍스트 부재 시 placeholder 또는 OCR 결과 입력"*).
- 사진-only 입력 시 서버가 `raw_text="(사진 입력)"` 한국어 fixed string placeholder로 자동 fallback — 클라이언트는 *송신만* 책임. Story 2.3 OCR이 `parsed_items` 추출 후 `raw_text`를 *"짜장면 1인분, 군만두 4개"* 형식 결과로 덮어씀(PATCH 또는 동시 INSERT — Story 2.3 spec에서 결정).
- placeholder 한국어 fixed string 채택 — i18n은 본 스토리 OUT (NFR-L2 한국어 1차 정합). 영문 변환은 *외주 인수 클라이언트* 시점(Story 8 개정 SOP).
- `image_key` UNIQUE 제약 X — 같은 사용자가 같은 사진을 여러 식단에 첨부 가능(*동일 메뉴 반복 식사* 시 사진 재사용). soft-deleted meal의 image_key가 R2에 잔존하는 GC 흐름은 Story 5.2 책임.

**R2 presigned URL 발급 — `app/adapters/r2.py` 신규 / boto3 S3-compat 패턴**

- architecture line 695 명시 `app/adapters/r2.py` — Cloudflare R2 (presigned URL 발급). 본 스토리가 첫 도입.
- boto3 S3 client에 R2 endpoint 주입 — `endpoint_url=f"https://{account_id}.r2.cloudflarestorage.com"`, `region_name="auto"`, `signature_version="s3v4"`(R2 필수). dev/test/CI 환경에서 boto3 client 생성은 *I/O 무발생*(서명 계산만) — 실제 R2 도달 X (단위 테스트 안전).
- presigned URL 5분 만료(`ExpiresIn=300`) — UX 정합(*촬영 → 즉시 업로드*) + 보안(만료 짧음). R2 측 `Content-Length-Range` policy는 sigv4 PUT presigned URL에서 *제한적 지원* — 서버 측 1차 게이트(Pydantic Field `le=10*1024*1024`)로 충분, R2 측 enforcement는 Story 8 hardening.
- image_key 형식 `meals/{user_id}/{uuid4}.{ext}` — user-scoped prefix는 (a) cross-user image_key enumeration 차단(공격자가 random key 시도해도 user_id mismatch), (b) R2 GC를 user_id 단위 prefix delete로 단순화(Story 5.2), (c) 인덱스 정렬 시 user 단위 grouping.

**R2 read URL — public CDN 직접 (signed read URL 도입은 Story 8 hardening)**

- architecture line 868: *"R2: 이미지 write는 백엔드 presigned URL, read는 CDN 직접 (인증 토큰 불필요한 public read)"*. 본 스토리 baseline은 이를 그대로 적용.
- `r2_public_base_url` 환경변수에 Cloudflare custom domain 또는 `*.r2.dev` URL 주입. 본 스토리 baseline은 *MVP 영업 데모 한정* — 식단 사진은 user PII이지만 *unguessable UUID 키 + user_id prefix*로 enumeration 사실상 불가(공격자가 동시에 user_id + uuid 매핑 정확히 추측해야 함).
- **trade-off 명시**: signed read URL(short-lived presigned GET) 도입은 (a) 백엔드 round trip 추가(image 표시 시점에 presign 호출), (b) 캐싱 complexity 증가(presigned URL 캐시 vs 만료), (c) UX 지연. MVP는 public read 채택, 외주 인수 클라이언트가 *민감도 高 도메인* 전환 시 Story 8 hardening 또는 W7 deploy 시점에 일괄 변경(deferred 등재).
- `r2_public_base_url` 미설정 시 fallback `https://<account_id>.r2.dev/<bucket>/<key>` — Cloudflare 표준. dev/CI 환경에서는 미설정 OK(환경변수 unset → 503 fail-fast at presign time, 클라이언트는 *오프라인 메시지* 표시).

**`POST /v1/meals/images/presign` 분리 라우터 + 단일 endpoint (forward Story 2.3 OCR 흡수 prep)**

- 신규 파일 `api/app/api/v1/meals_images.py` — Story 2.1 `meals.py` 단일 파일 size 한계 + forward Story 2.3(`POST /v1/meals/images/{image_key}/parse` OCR endpoint) 흡수 prep.
- prefix `/v1/meals/images` — endpoint `POST /presign`. forward Story 2.3 endpoint는 같은 라우터에 `POST /{image_key}/parse` 형태로 추가 가능.
- 1파일 1 router 패턴(architecture line 544) 정합 — *meals_images*는 `meals.py`의 *서브 라우터*가 아닌 별도 모듈이며 `app/main.py`에서 같은 prefix하에 등록.

**`MealCreateRequest`/`MealUpdateRequest` invariant — *최소 1 필드(raw_text 또는 image_key)* + ownership 검증**

- POST: `raw_text=None and image_key=None` 거부 → 400 + *"at least one of raw_text, image_key required"* (Pydantic `@model_validator`).
- PATCH: `raw_text/ate_at/image_key` 셋 다 absent 거부 → 400 + *"at least one field required"* (Story 2.1 정합 갱신).
- ownership: `image_key.startswith(f"meals/{user.id}/")` 검증 → cross-user image_key 도용 차단 (사용자 A가 사용자 B의 image_key를 자기 식단에 첨부 시 400 + `code=meals.image.foreign_key_rejected`). image_key=None(PATCH 클리어)은 검증 대상 X.
- ownership 검증 위치 — Pydantic field_validator는 user context 미접근 → 라우터 inline 검증 (Story 2.1 race-free 패턴 변경 0).
- PATCH 후 *raw_text/image_key 둘 다 None* invariant — `raw_text` DB NOT NULL + Story 2.1 D4(`raw_text=null` 송신은 no-op) 시맨틱으로 SQL 레벨에서 unreachable. 별도 가드 도입 X (YAGNI, 추가 SELECT round-trip 0).

**Mobile expo-image-picker 단일 라이브러리 + permissions wrapper hook + PermissionDeniedView 표준 컴포넌트**

- `expo-image-picker`는 Expo SDK 54 표준 라이브러리 — 카메라/갤러리 양쪽 흡수 + 권한 hook 제공. `expo-camera`/`expo-media-library` 별도 도입 회피(yagni — 본 스토리는 *사진 1장 선택* 한정, 카메라 미리보기/필터/실시간 영상은 OUT).
- `mobile/lib/permissions.ts` wrapper — (a) 모듈 1지점 변경, (b) 향후 `expo-permissions` deprecation 대응 단순화, (c) 테스트 모킹 1지점.
- `PermissionDeniedView`는 Story 4.1 알림 권한도 동일 props로 재사용 가능 — *Epic 2 사진 한정* baseline + forward props design.
- 권한 요청 시점 — *첫 사진 입력 시점*에만 prompt(앱 진입 시 미요청 — UX 표준 / iOS App Store 정책 정합).

**모바일 R2 직접 업로드 — 백엔드 proxy 회피 (메모리 폭발 방지)**

- 클라이언트가 백엔드에 base64 또는 multipart/form-data로 이미지 송신 시 백엔드 메모리 + Railway egress 비용 폭발 — architecture line 868 정합 *직접 업로드*.
- 클라이언트 흐름: (a) `POST /presign` 호출 → upload_url 수신, (b) RN `fetch(upload_url, { method: 'PUT', body: blob })` → R2 직접 PUT, (c) 성공 시 `image_key`를 `POST /meals` body에 첨부.
- RN `fetch`는 file URI를 `Blob`으로 변환 가능 — `await fetch(uri).then(r => r.blob())`. iOS HEIC/HEIF는 native 변환 미요구(R2가 raw bytes 보존 — 표시 시점에 브라우저/모바일 OS가 decode).

**오프라인 사진 입력 OUT — 사진은 온라인 필수 (epic 2.5 명시 정합)**

- epic AC line 575: *"사진 입력은 오프라인 큐잉 대상 아님 — 오프라인에서 사진 시도 시 '사진은 온라인에서 처리 가능 — 텍스트 입력 또는 온라인 시 다시 시도' 안내"*.
- 본 스토리 baseline은 *오프라인 감지 X* — 사진 업로드 실패 시 *네트워크 실패*로 분류 → Alert + 재시도. 명시적 *오프라인 모드 안내*는 Story 2.5(오프라인 텍스트 큐잉)이 NetInfo 통합 시점에 흡수.

### Anti-pattern 카탈로그 (본 스토리 영역)

[Source: Story 2.1 Anti-pattern 카탈로그, architecture.md#Anti-pattern (line 557-565)]

- ❌ Backend 이미지를 base64 또는 multipart로 백엔드에 POST — 메모리 폭발 + Railway egress 비용. **채택**: R2 직접 업로드.
- ❌ Backend `r2.py` adapter에서 매 호출마다 `boto3.client(...)` 신규 생성 — 50ms 오버헤드. **채택**: 모듈 lazy singleton.
- ❌ Backend `image_key` UNIQUE 제약 — 같은 사진 여러 식단 첨부 차단(사용성 저하). **채택**: 제약 X, GC는 Story 5.2.
- ❌ Backend `meals.image_key NOT NULL` — Story 2.1 텍스트-only 호환 깨짐. **채택**: nullable.
- ❌ Backend `raw_text NULL 허용` — Story 2.1 invariant 깸 + Story 2.3 OCR fallback 흐름 복잡화. **채택**: NOT NULL 유지 + 서버 placeholder fallback.
- ❌ Backend cross-user `image_key` 도용 허용 — privacy leak. **채택**: ownership 검증(`startswith(f"meals/{user.id}/")`).
- ❌ Backend signed read URL 도입 — MVP 영업 데모 한정 + UX 지연 + 캐싱 complexity. **채택**: public read CDN URL (Story 8 hardening 시 재검토).
- ❌ Backend `POST /v1/meals/{id}/image` (PRD line 705) endpoint 도입 — 사용자가 *meal 생성 전* 이미지 업로드 흐름이 자연스러움. **채택**: `POST /v1/meals/images/presign` 분리(meal id 사전 생성 회피).
- ❌ Backend `MealCreateRequest`에 `image_url` 필드 직접 수용 — 클라이언트가 임의 URL 송신 시 외부 image proxy 악용 가능. **채택**: `image_key`만 수용, `image_url`은 *응답 derive*.
- ❌ Backend `expires_at`을 Python `datetime.now() + timedelta(minutes=5)`로 계산 — clock skew 차단. **채택**: boto3 SDK 자체가 reliable expiry 시계 — `datetime.now(UTC) + timedelta(seconds=300)` 동등 안전(boto3 sign 시점이 server local clock와 동일).
- ❌ Mobile `expo-camera` + `expo-media-library` 별도 도입 — `expo-image-picker`로 충분. yagni.
- ❌ Mobile 커스텀 카메라 화면(`CameraView.tsx`) — OS 표준 picker로 충분. yagni.
- ❌ Mobile 권한 거부 시 빈 화면 — epic AC 명시 *"빈 화면 노출 금지"*. **채택**: `PermissionDeniedView` 표준 컴포넌트.
- ❌ Mobile 권한 hook을 컴포넌트마다 inline 호출 — DRY 위반. **채택**: `mobile/lib/permissions.ts` wrapper.
- ❌ Mobile 사진 미리보기에 `expo-image` 도입 — RN 표준 `Image`로 충분. yagni (blurhash/캐시 우선은 Story 8 polish).
- ❌ Mobile R2 PUT에 `XMLHttpRequest` + 진행률 UI 도입 — 단순 spinner로 충분. yagni.
- ❌ Mobile `AbortController` 미적용 — 30초 timeout 강제로 stuck upload 방지.
- ❌ Mobile 사진 첨부 후 *제거* 액션을 *clear* 명시 송신 X — PATCH 모드에서 `image_key=null` 명시 송신 vs absent 차이가 시맨틱 변경. **채택**: `imageClearedExplicitly` state + PATCH `body.image_key = null` 명시.
- ❌ Mobile EXIF 위치/datetime 보존 — 사용자 위치 leak. **deferred**: Story 8 privacy hardening (`expo-image-picker.quality`만으로는 EXIF 미제거 — `expo-image-manipulator` 통합 시점).
- ❌ Mobile `mealCreateSchema`(zod)에 `superRefine` 도입 — schema 복잡도. **채택**: 부모 컴포넌트(`input.tsx`)에서 *raw_text or image_key 최소 1* 검증 inline.
